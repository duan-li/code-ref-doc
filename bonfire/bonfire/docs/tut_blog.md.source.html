   1  # A Simple Blog Tutorial
   2  
   3  This tutorial will teach you the basics of how to work with Bonfire. Along the way, you will learn the basic concepts you need to know to best work within the system. You will learn the details and the code. We believe it's best to learn how to do things the long way, before using the power of the [Module Builder](using_module_builder).
   4  
   5  <a name="newmodule"></a>
   6  ## Creating A Module
   7  
   8  While modules are not required to use Bonfire, it makes extensive use of modules to encourage code reuse and organization. Whithin this tutorial we will create a new blog module.
   9  
  10  The first step is to get the folder structure created and make sure the settings are correct for it to show up in the admin menu.
  11  
  12  ### Folder Structure
  13  
  14  Within the <tt>application/modules</tt> folder, create a new folder named <tt>blog</tt>, and create the following subfolders: assets, config, controllers, migrations, models, views. When you're done your module should look like:
  15  
  16      bonfire/
  17          modules/
  18              blog/
  19                  assets/
  20                  config/
  21                  controllers/
  22                  migrations/
  23                  models/
  24                  views/
  25  
  26  Not every module will require every folder, but these are the basic folders every module can use.
  27  
  28  ### Basic Configuration
  29  
  30  To make sure that your blog shows up in the admin area, we need to create a config file that holds various settings that let Bonfire know about your module. The config file is not required, but allows more control over how your module appears throughout the system.
  31  
  32  Create a new file in your new <tt>config</tt> folder, called <tt>config.php</tt>.
  33  
  34  ```
  35  <?php if ( ! defined('BASEPATH')) exit('No direct script access allowed');
  36      $config['module_config'] = array(
  37          'name'          => 'Blog',
  38          'description'   => 'A Simple Blog Example',
  39          'author'        => 'Your Name',
  40          'homepage'      => 'http://...',
  41          'version'       => '1.0.1',
  42          'menu'          => array(
  43              'context'   => 'path/to/view'
  44          ),
  45          'weights'       => array(
  46              'context'   => 0
  47          )
  48      );
  49  ```
  50  
  51  
  52  Not all of these settings will be used for the Blog module, but you should understand how they work.
  53  
  54  **Name** is the human-readable name of your module.
  55  
  56  **Description** is a short description of your module and will appear in a list of installed modules.
  57  
  58  **Author** is your name or the name of your organization/team.
  59  
  60  **Homepage** is the URL to the homepage of your module. Will appear as a link in the installed modules page.
  61  
  62  **Version** is a simple version string.
  63  
  64  **Menu** is an array that lets you specify sub-menus that will appear under each context. It points to a view file within your module that contains a &lt;ul&gt; of your options.
  65  
  66  **Weights** is an array that allows you to order the menu items within the main context navigation. Think of it like the menu is a bowl of water. The heavier the weight (or the larger the value), the farther down the menu it will sink.
  67  
  68  ### Setup the Database
  69  
  70  To allow for simple versioning of your database we will use Bonfire's migrations. This is great for working in teams where modifications to the database might be made by several members on the development team. It also makes it fairly simple to bring your changes to the production database. In the case of a module like our blog module, it allows us to easily re-use the module in another application and quickly get the database setup.
  71  
  72  Migrations are simply a set of commands that are run to make changes to the database or remove changes. They are stored in the module's migrations folder, using sequentially numbered files.
  73  
  74  Create a new file at <tt>blog/migrations/001_Initial_tables.php</tt>.
  75  
  76  ```php
  77  class Migration_Initial_tables extends Migration
  78  {
  79  
  80      public function up()
  81      {
  82          $this->load->dbforge();
  83  
  84          $this->dbforge->add_field('post_id BIGINT(20) UNSIGNED NOT NULL AUTO_INCREMENT');
  85          $this->dbforge->add_field('title VARCHAR(255) NOT NULL');
  86          $this->dbforge->add_field('slug VARCHAR(255) NOT NULL');
  87          $this->dbforge->add_field('body TEXT NULL');
  88          $this->dbforge->add_field('created_on DATETIME NOT NULL');
  89          $this->dbforge->add_field('modified_on DATETIME NULL');
  90          $this->dbforge->add_field('deleted TINYINT(1) NOT NULL DEFAULT 0');
  91          $this->dbforge->add_key('post_id', TRUE);
  92  
  93          $this->dbforge->create_table('posts');
  94  
  95          // Create the Permissions
  96          $this->load->model('permission_model');
  97          $this->permission_model->insert(array(
  98              'name'          => 'Bonfire.Blog.View',
  99              'description'   => 'To view the blog menu.',
 100              'status'        => 'active'
 101          ));
 102  
 103          // Assign them to the admin role
 104          $this->load->model('role_permission_model');
 105          $this->role_permission_model->assign_to_role('Administrator', 'Bonfire.Blog.View');
 106      }
 107  
 108      //--------------------------------------------------------------------
 109  
 110      public function down()
 111      {
 112          $this->load->dbforge();
 113  
 114          $this->dbforge->drop_table('posts');
 115      }
 116  
 117      //--------------------------------------------------------------------
 118  
 119  }
 120  ```
 121  
 122  The <tt>up()</tt> method is ran whenever this migration is 'installed'. The <tt>down()</tt> method is ran whenever the migration is 'un-installed'.
 123  
 124  We use CodeIgniter's <tt>dbforge</tt> class here to create the table, but you could use the <tt>database</tt> class to run raw queries, if you wanted to. We also load up the <tt>permissions_model</tt> and <tt>role_permission_model</tt> and create a new permission, <tt>Bonfire.Blog.View</tt> and assigns the Admin role that permission so you can actually see the blog page in the Content menu.
 125  
 126  To install this migration, navigate to Developer / Database / Migrations. Click on the Modules tab. You will see your Blog module shows up there. Select the <tt>001_Initial_tables.php</tt> from the dropdown, and click 'Migrate Module'. Your <tt>up()</tt> method will run, installing the posts table into your database. Until we create the controller, though, it will not show up in the menu.

 128  ### Content Context
 129  
 130  In order to manage your blog, we will create a new entry under the Content menu that takes us to all of the blog management features. To get started we will need one controller and one view.
 131  
 132  Create a new controller, <tt>blog/controllers/content.php</tt>. Each context uses a controller of the same name. In this case we want to create some actions for the Content Context, so we create a controller named <tt>content.php</tt>.
 133  
 134  ```php
 135      class Content extends Admin_Controller
 136      {
 137  
 138          public function __construct()
 139          {
 140              parent::__construct();
 141  
 142              Template::set('toolbar_title', 'Manage Your Blog');
 143          }
 144  
 145          //--------------------------------------------------------------------
 146  
 147          public function index()
 148          {
 149              Template::render();
 150          }
 151  
 152          //--------------------------------------------------------------------
 153  
 154      }
 155  ```
 156  
 157  Notice that the class is named the same as the Context and it extends <tt>Admin_Controller</tt>. The Admin_Controller is one of several controllers provided by Bonfire to take care of a few functions for you. In this case, the controller makes sure that we are logged in, sets up pagination defaults, sets the theme to the admin theme, and gets our form_validation library loaded and setup so that it works properly with HMVC.
 158  
 159  In the <tt>__construct()</tt> method we are doing one thing currently, and that is to set the text that shows up in the sub-navigation bar just below the main menu in the admin area. We'll make more use of this bar in the future.
 160  
 161  The <tt>index()</tt> method is the method that will be called by default when you click on the <tt>Blog</tt> menu item. We use the Template library's <tt>render()</tt> method to display the view for this page. By default, this will look for a view in the module's views folder under the context name and method name. In this case, it would be searching for <tt>blog/views/content/index.php</tt>. Create that file now.
 162  
 163  
 164  ```
 165      <h1>Blog Index</h1>
 166  ```
 167  
 168  We are keeping it simple just to make sure everything is in working order for now.
 169  
 170  Save the files, then navigate to **http://yoursite.com/admin/content/blog** to view your new page. You can also refresh the screen now, and a link will show up in the Content menu that you can use.

 172  ### The Blog Model
 173  
 174  In order to view any posts, we will need to create a model that interacts with the database for us. Thanks to Bonfire's <tt>MY_Model</tt> base class, this is a very simple task.
 175  
 176  Create a new model file at <tt>blog/models/post_model.php</tt>.
 177  
 178  ```php
 179      class Post_model extends MY_Model
 180      {
 181  
 182          protected $table_name   = 'posts';
 183          protected $key          = 'post_id';
 184          protected $set_created  = true;
 185          protected $set_modified = true;
 186          protected $soft_deletes = true;
 187          protected $date_format  = 'datetime';
 188  
 189          //---------------------------------------------------------------
 190  
 191      }
 192  ```
 193  
 194  This is everything needed to get some pretty flexible CRUD setup and running for your post model. There are a lot more options available in the full [model file](bonfire_models) but this is all we need for now.
 195  
 196  **$table_name** is the name of the database table that the data is stored in.
 197  
 198  **$key** is the name of the primary key the table uses.
 199  
 200  **$set_created** tells the system whether it should automatically store the date the object was created. In order for this to work, you must have a <tt>created_on</tt> datetime field in your table.
 201  
 202  **$set_modified** tells the system whether it should automatically store the date the object was last modified. In order for this to work, you must have a <tt>modified_on</tt> datetime field in your table.
 203  
 204  **$soft_deletes** tells the system whether a delete function should permanently delete the row (a 'hard' delete), or simply set a deleted flag in the table. This requires that you have a <tt>deleted</tt> tinyint(1) field in your table.
 205  
 206  With the settings above, our post_model will:
 207  
 208  * Store our data in the <tt>posts</tt> table in our database.
 209  * Each row will have a primary key called <tt>post_id</tt>.
 210  * Store the date the row was created in the <tt>created_on</tt> field.
 211  * Store the date the row was last modified in the <tt>modified_on</tt> field.
 212  * Set the <tt>deleted</tt> field to 1 when a row is deleted, instead of permanently removing it.
 213  
 214  Since the post_model will be used in nearly every method in our content controller, we will autoload it in the controller's <tt>__construct()</tt> method.
 215  
 216  ```php
 217      public function __construct()
 218      {
 219          parent::__construct();
 220  
 221          $this->load->model('post_model');
 222  
 223          Template::set('toolbar_title', 'Manage Your Blog');
 224      }
 225  ```

 228  ### Listing Posts in Admin
 229  
 230  It's now time to expand our blog's content index method to show us a list of all posts in the system. Add the following lines to the <tt>index()</tt> method of the <tt>content</tt> controller, before the <tt>Template::render()</tt> method.
 231  
 232  ```php
 233      $posts = $this->post_model->where('deleted', 0)->find_all();
 234  
 235      Template::set('posts', $posts);
 236  ```
 237  
 238  This calls our <tt>post_model</tt>'s <tt>find_all()</tt> method to retrieve all posts in the system. However, we don't want any deleted posts, so we filter those out with the <tt>where()</tt> method.
 239  
 240  We then use the Template class' <tt>set()</tt> method to make the data available to our view. In the view, this can be accessed as the <tt>$posts</tt> variable.
 241  
 242  Edit your index.php view file to reflect the following:
 243  
 244  ```
 245      <div class="admin-box">
 246          <h3>Blog Posts</h3>
 247  
 248          <?php echo form_open(); ?>
 249  
 250              <table class="table table-striped">
 251                  <thead>
 252                      <tr>
 253                          <th class="column-check"><input class="check-all" type="checkbox" /></th>
 254                          <th>Title</th>
 255                          <th style="width: 10em">Date</th>
 256                      </tr>
 257                  </thead>
 258                  <tfoot>
 259                      <tr>
 260                          <td colspan="3">
 261                              With selected:
 262                              <input type="submit" name="submit" class="btn" value="Delete">
 263                          </td>
 264                      </tr>
 265                  </tfoot>
 266                  <tbody>
 267                  <?php if (isset($posts) && is_array($posts)) :?>
 268                      <?php foreach ($posts as $post) : ?>
 269                      <tr>
 270                          <td><input type="checkbox" name="checked[]" value="<?php echo $post->post_id ?>" /></td>
 271                          <td>
 272                              <a href="<?php echo site_url(SITE_AREA .'/content/blog/edit_post/'. $post->post_id) ?>">
 273                                  <?php e($post->title); ?>
 274                              </a>
 275                          </td>
 276                          <td>
 277                              <?php echo date('M j, Y g:ia'); ?>
 278                          </td>
 279                      </tr>
 280                      <?php endforeach; ?>
 281                  <?php else: ?>
 282                      <tr>
 283                          <td colspan="3">
 284                              <br/>
 285                              <div class="alert alert-warning">
 286                                  No Posts found.
 287                              </div>
 288                          </td>
 289                      </tr>
 290                  <?php endif; ?>
 291                  </tbody>
 292              </table>
 293  
 294          <?php echo form_close(); ?>
 295      </div>
 296  ```
 297  
 298  This creates a table that will list each blog post, if any exist. If they don't then a notice will be displayed.
 299  
 300  Most of this should be self-explanatory, but there is one new function nestled in there, <tt>e()</tt>. This method is a convenience method that you should consider using wherever you are displaying user-entered data. It simply echos out the string, using the <tt>htmlentities()</tt> function to help protect against XSS and CSRF attacks.

 302  ### Module Sub-Menus
 303  
 304  Now we just need a way to create new posts. Let's start by creating a new sub-menu that allows us to access other pages. This is not intended for long menus, but to provide a short list of major areas within your module. You will see this used throughout Bonfire, and it appears on the right side of the page, just under the main menu. This is the same bar that holds your <tt>$toolbar_title</tt>.
 305  
 306  First, we create a view file holds the menu itself. Create a new file at <tt>blog/views/content/sub_nav.php</tt>.
 307  
 308  ```php
 309      <ul class="nav nav-pills">
 310          <li <?php echo $this->uri->segment(4) == '' ? 'class="active"' : '' ?>>
 311              <a href="<?php echo site_url(SITE_AREA .'/content/blog') ?>">Posts</a>
 312          </li>
 313          <li <?php echo $this->uri->segment(4) == 'create' ? 'class="active"' : '' ?>>
 314              <a href="<?php echo site_url(SITE_AREA .'/content/blog/create') ?>">New Post</a>
 315          </li>
 316      </ul>
 317  ```
 318  
 319  Each link in this list will take us to a method within our content controller. We also check the url to see if this is an active link or not.
 320  
 321  To make this menu show up, we need to add it to our content controller's <tt>__construct()</tt> method.
 322  
 323  ```php
 324      public function __construct()
 325      {
 326          parent::__construct();
 327  
 328          $this->load->model('post_model');
 329  
 330          Template::set('toolbar_title', 'Manage Your Blog');
 331          Template::set_block('sub_nav', 'content/sub_nav');
 332      }
 333  ```
 334  
 335  Reload your page in the admin area, and you will see the new menu appear. Clicking on 'New Post' throws an error since we haven't created that method yet. We will do that next.

 337  ### Create A Post
 338  
 339  We will start things simple by just displaying the form to create a new post, then deal with saving the post later.
 340  
 341  Create a new <tt>create()</tt> method in your content controller.
 342  
 343  ```php
 344      public function create()
 345      {
 346          Template::set('toolbar_title', 'Create New Post');
 347          Template::set_view('content/post_form');
 348          Template::render();
 349      }
 350  ```
 351  
 352  This sets the <tt>toolbar_title</tt> of the page, says that we want to use the view named <tt>views/content/post_form.php</tt>, and renders the form.
 353  
 354  Now we need create the form itself. We're using a file called <tt>post_form</tt> because we want to be able to use the form for both the create and edit pages.
 355  
 356  ```
 357      <div class="admin-box">
 358          <h3>New Post</h3>
 359  
 360          <?php echo form_open(current_url(), 'class="form-horizontal"'); ?>
 361  
 362              <div class="control-group <?php if (form_error('title')) echo 'error'; ?>">
 363                  <label for="title">Title</label>
 364                  <div class="controls">
 365                      <input type="text" name="title" class="input-xxlarge" value="<?php echo isset($post) ? $post->title : set_value('title'); ?>" />
 366                      <?php if (form_error('title')) echo '<span class="help-inline">'. form_error('title') .'</span>'; ?>
 367                  </div>
 368              </div>
 369  
 370              <div class="control-group <?php if (form_error('slug')) echo 'error'; ?>">
 371                  <label for="slug">Slug</label>
 372                  <div class="controls">
 373                      <div class="input-prepend">
 374                          <span class="add-on"><?php echo site_url() .'/blog/' ?></span>
 375                          <input type="text" name="slug" class="input-xlarge" value="<?php echo isset($post) ? $post->slug : set_value('slug'); ?>" />
 376                      </div>
 377                      <?php if (form_error('slug')) echo '<span class="help-inline">'. form_error('slug') .'</span>'; ?>
 378                      <p class="help-block">The unique URL that this post can be viewed at.</p>
 379                  </div>
 380              </div>
 381  
 382              <div class="control-group <?php if (form_error('body')) echo 'error'; ?>">
 383                  <label for="body">Content</label>
 384                  <div class="controls">
 385                      <?php if (form_error('body')) echo '<span class="help-inline">'. form_error('body') .'</span>'; ?>
 386                      <textarea name="body" class="input-xxlarge" rows="15"><?php echo isset($post) ? $post->body : set_value('body') ?></textarea>
 387                  </div>
 388              </div>
 389  
 390              <div class="form-actions">
 391                  <input type="submit" name="submit" class="btn btn-primary" value="Save Post" />
 392                  or <a href="<?php echo site_url(SITE_AREA .'/content/blog') ?>">Cancel</a>
 393              </div>
 394  
 395          <?php echo form_close(); ?>
 396      </div>
 397  ```
 398  
 399  Most of this is straight-forward. I do want to point out that for the values, we are checking whether a <tt>$post</tt> value is set or not, and then uses the form helper's <tt>set_value()</tt> method to set the value in the case of errors on the form.
 400  
 401  We also use the form_validation library's <tt>form_error()</tt> function to setup individual errors for each field.

 403  <b>Saving the Post</b>
 404  
 405  Now, let's actually make it functional. In your post_model, we need to let it know what validation rules to use during both inserts or updates. Add the following class variable to the post_model:
 406  
 407  ```
 408      protected $validation_rules = array(
 409          array(
 410              'field' => 'title',
 411              'label' => 'Title',
 412              'rules' => 'trim|strip_tags|xss_clean'
 413          ),
 414          array(
 415              'field' => 'slug',
 416              'label' => 'Slug',
 417              'rules' => 'trim|strip_tags|xss_clean'
 418          ),
 419          array(
 420              'field' => 'body',
 421              'label' => 'Body',
 422              'rules' => 'trim|strip_tags|xss_clean'
 423          )
 424      );
 425  ```
 426  
 427  These rules follow the same format as the [form validation library](http://ellislab.com/codeigniter/user-guide/libraries/form_validation.html#validationrulesasarray). The one thing to be aware of is that these rules are used for both inserts and updates. This can cause a problem with required fields on inserts. We'll add the <tt>insert_validation_rules</tt> class variable to the pose_model to provide any additional rules we want applied during an insert only.
 428  
 429  ```php
 430      protected $insert_validation_rules = array(
 431          'title' => 'required',
 432          'body'  => 'required'
 433      );
 434  ```
 435  
 436  Now, whenever you do an insert or an update, the data is run through the form validation library. If it fails validation, the model will return FALSE and your controller can respond accordingly.
 437  
 438  Now then modify the controller's <tt>create()</tt> method to actually save the data:
 439  
 440  ```php
 441      public function create()
 442      {
 443          if ($this->input->post('submit'))
 444          {
 445              $data = array(
 446                  'title' => $this->input->post('title'),
 447                  'slug'  => $this->input->post('slug'),
 448                  'body'  => $this->input->post('body')
 449              );
 450  
 451              if ($this->post_model->insert($data))
 452              {
 453                  Template::set_message('You post was successfully saved.', 'success');
 454                  redirect(SITE_AREA .'/content/blog');
 455              }
 456          }
 457  
 458          Template::set('toolbar_title', 'Create New Post');
 459          Template::set_view('content/post_form');
 460          Template::render();
 461      }
 462  ```

 464  ### Editing Posts
 465  
 466  Editing our posts is very simple to do now. Simply add the following <tt>edit_post()</tt> method to your controller and you're up and running:
 467  
 468  ```php
 469      public function edit_post($id=null)
 470      {
 471          if ($this->input->post('submit'))
 472          {
 473              $data = array(
 474                  'title' => $this->input->post('title'),
 475                  'slug'  => $this->input->post('slug'),
 476                  'body'  => $this->input->post('body')
 477              );
 478  
 479              if ($this->post_model->update($id, $data))
 480              {
 481                  Template::set_message('You post was successfully saved.', 'success');
 482                  redirect(SITE_AREA .'/content/blog');
 483              }
 484          }
 485  
 486          Template::set('post', $this->post_model->find($id));
 487  
 488          Template::set('toolbar_title', 'Edit Post');
 489          Template::set_view('content/post_form');
 490          Template::render();
 491      }
 492  ```

 494  ## The Public Context
 495  
 496  Now that we have basic administration pages in place, it's time to actually let the users view your awesome blog posts. This requires that we create a new controller, in the same blog module, called <tt>blog</tt>. This will handle what we call your <em>Public Context</em> and is simply a front-facing controller that will directly map to the URI. In this case, you can view this controller at <tt>http://yoursite.com/blog</tt>.
 497  
 498  Create a new file, <tt>modules/blog/controllers/blog.php</tt>
 499  
 500  ```php
 501      class Blog extends Front_Controller
 502      {
 503  
 504          public function __construct()
 505          {
 506              parent::__construct();
 507  
 508              $this->load->model('post_model');
 509          }
 510  
 511          //--------------------------------------------------------------------
 512  
 513          public function index()
 514          {
 515              $this->load->helper('typography');
 516  
 517              $posts = $this->post_model->order_by('created_on', 'asc')
 518                                        ->limit(5)
 519                                        ->find_all();
 520  
 521              Template::set('posts', $posts);
 522  
 523              Template::render();
 524          }
 525  
 526          //--------------------------------------------------------------------
 527  
 528      }
 529  ```
 530  
 531  This is just a typical CodeIgniter controller, which means that anything you can do in straight CodeIgniter, you can do here.
 532  
 533  In this case, we're loading our model in the constructor, since we know that we'll be using it in every method. Then we create an index method to list the 5 most recent posts.
 534  
 535  Then, it looks for a view file at <tt>blog/views/index.php</tt>. Create that file now.
 536  
 537  ```php
 538          <?php foreach ($posts as $post) :?>
 539          <div class="post">
 540              <h2><?php e($post->title) ?></h2>
 541  
 542              <?php echo auto_typography($post->body) ?>
 543          </div>
 544          <?php endforeach; ?>
 545  
 546      <?php else : ?>
 547          <div class="alert alert-info">
 548              No Posts were found.
 549          </div>
 550      <?php endif; ?>
 551  ```
 552  
 553  This is a very simple view, but should be enough to give you an understanding.
 554  
 555  Navigate to <tt>http://yoursite.com/index.php/blog</tt> and you should see the 5 most recent blog posts for you.
