   1  ## Reporting an Issue
   2  
   3  Bonfire uses [GitHub Issue Tracking](https://github.com/ci-bonfire/Bonfire/issues) to track issues (primarily bugs and new code contributions). If you find a bug in Bonfire, this is the place to start. You will need a (free) [GitHub](http://github.com) account in order to help out.
   4  
   5  <a name="creating-bug-reports"></a>
   6  ### Creating a Bug Report
   7  
   8  If you have found a problem in Bonfire start by searching the existing issues at [GitHub](https://github.com/ci-bonfire/Bonfire/issues) to make sure it hasn't already been reported. If there is not existing issue you can [add a new one](https://github.com/ci-bonfire/Bonfire/issues/new).
   9  
  10  At a minimum, your issue report needs a title and a description of the problem. But that's not all, that's just the minimum. You should include a code sample, if applicable, and the steps needed to recreate. It is also recommended to include your PHP version, and operating system. The goal is to make it easy for yourself--or others--to recreate and fix the bug.
  11  
  12  You should also assign one of the labels to the issue. For most issues, simply assigning the 'Bug' label will be all you need to do. If you have found a truly critical bug that stops the execution of the program and makes it completely unusable (as opposed to just an annoyance or part not working) you can assign the 'Critical' label to it. Please be considerate of the developers, though, and don't use this label unless it is absolutely a show stopper.
  13  
  14  Once you've submitted your report, don't hold your breath for a fix. Unless this is flagged as a "Critical, It Won't Work At All!" type of bug, you are creating the issue in the hopes that other developers will work with you to find a fix. There is no guarantee that someone else will jump in a fix the problem for you. Creating an issue like this is mostly to help yourself start on the path of fixing the problem and for others to confirm it with a "I'm having this problem too" comment.
  15  
  16  
  17  <a name="feature-requests"></a>
  18  ### Submitting Feature Requests
  19  
  20  You can submit a feature request in much the same was reporting a bug (above). This time, however, you need to make sure to add a 'Feature Request' label to the issue so that everyone can easily sort the issues.
  21  
  22  
  23  <a name="fix-issues"></a>
  24  ## Helping Fix Existing Issues
  25  
  26  The next step beyond reporting issues is to help the core team resolve existing issues. If you check the [Everyone's Issues](https://github.com/ci-bonfire/Bonfire/issues) list in GitHub, you're sure to find issues that already need resolving. Here's how you--regardless of your technical expertise--can help out.
  27  
  28  <a name="verify-bugs"></a>
  29  ### Verifying Bug Reports
  30  
  31  For starters, it helps to just verify bug reports. Can you reproduce the reported issue on your own computer? If so, you can add a comment to the issue saying that you're seeing the same thing.
  32  
  33  If something is very vague, can you help squish it down into something specific? Maybe you can provide additional information to help reproduce a bug, or eliminate needless steps that aren't required to help demonstrate the problem.
  34  
  35  Anything you can do to make bug reports more succinct or easier to reproduce is a help to folks trying to write code to fix those bugs--whether you end up writing the code yourself or not.
  36  
  37  <a name="testing-patches"></a>
  38  ### Testing Patches
  39  
  40  You can also help out by examining pull requests that have been submitted to Bonfire via GitHub. To apply someone's changes you need to first create a dedicated branch:
  41  
  42      $ git checkout -b testing_branch
  43  
  44  Then you can use their remote branch to update your codebase. For example, let's say the GitHub user JohnSmith has forked and pushed to the topic branch located at https://github.com/JohnSmith/Bonfire.
  45  
  46      $ git remote add JohnSmith git://github.com/JohnSmith/Bonfire.git
  47      $ git pull JohnSmith topic
  48  
  49  After applying their branch, test it out! Here are some things to think about:
  50  
  51  * Does the change actually work?
  52  * Are you happy with the tests? Can you follow what they're testing? Are there any tests missing?
  53  * Does it have proper documentation coverage? Should documentation elsewhere be updated?
  54  * Do you like the implementation? Can you think of a nicer or faster way to implement a part of their change?
  55  
  56  Once you're happy that the pull request contains a good change, comment on the GitHub issue indicating your approval. Your comment should indicate that you like the change and what you like about it. Something like:
  57  
  58  > I like the way you've restructured that code in generate_finder_sql, much nicer. The tests look good too.
  59  
  60  If your comment simply says "+1", then odds are that other reviewers aren't going to take it too seriously. Show that you took the time to review the pull request.

  63  <a name="bonfire-docs"></a>
  64  ## Contributing to Bonfire Documentation
  65  
  66  Bonfire has two main sets of documentation: the Guides help you learn Bonfire, while the API is a reference.
  67  
  68  You can help improve the guides by making them more coherent, adding missing information, correcting factual errors, fixing typos, bringing it up to date with the latest edge Bonfire. To get involved in the translation of Bonfire guides, please see [Translating Bonfire Guides]().
  69  
  70  Changes are made directly within this wiki so please double-check all edits for correctness and good English. If you are a non-native English speaker, feel free to post on the [forums](http://forums.cibonfire.com) or post an issue here on GitHub asking someone to look over your English for the portion you edited.
  71  
  72  If you are unsure of the documentation changes, you can create an issue in the Bonfire [issues tracker](https://github.com/ci-bonfire/Bonfire/issues?sort=created&direction=desc&state=open) on GitHub.
  73  
  74  When working with documentation, please take into account the [API Documentation Guidelines](api_documentation_guidelines.html) and the [Coding Conventions](coding_conventions).

  76  <a name="bonfire-code"></a>
  77  ## Contributing to the Bonfire Code
  78  
  79  <a name="clone-repo"></a>
  80  ### Clone the Bonfire Repository
  81  
  82  The first thing you need to do to be able to contribute code is to clone the repository:
  83  
  84      $ git clone git://github.com/ci-bonfire/Bonfire.git
  85  
  86  and create a dedicated branch:
  87  
  88      $ cd Bonfire
  89      $ git checkout -b my_new_branch
  90  
  91  It doesn't really matter what name you use, because this branch will only exist on your local computer.
  92  
  93  <a name="write-code"></a>
  94  ### Write Your Code
  95  
  96  Now get busy and add or edit code. You're on your branch now, so you can write whatever you want (you can check to make sure you're on the right branch with git branch -a). But if you're planning to submit your change back for inclusion in Bonfire, keep a few things in mind:
  97  
  98  * Get the code right
  99  * Use Bonfire idioms and classes
 100  * Include tests that fail without your code, and pass with it
 101  * Update the documentation, the surrounding one, examples elsewhere, guides, whatever is affected by your contribution
 102  
 103  <a name="follow-conventions"></a>
 104  ### Follow the Coding Conventions
 105  
 106  Bonfire follows a simple set of [Coding Conventions](coding_conventions) which are summarized here, but are given a more complete treatment in their own guide.
 107  
 108  * Spaces, not tabs, are preferred.
 109  * No trailing whitespace. Blank lines should not have any space.
 110  * Prefer && / || over AND / OR.
 111  * Always use brackets for foreach, if else, etc.
 112  * Brackets on their own lines.
 113  * Follow the conventions you see used in the source already.
 114  
 115  <a name="sanity-check"></a>
 116  ### Sanity Check
 117  
 118  You should not be the only person who looks at the code before you submit it. You know at least one other developer, right? Show them what you're doing and ask for feedback. Doing this in private before you push a patch out publicly is the "smoke test" for a patch: if you can't convince one other developer of the beauty of your code, you're unlikely to convince the core team either.
 119  
 120  <a name="commit-changes"></a>
 121  ### Commit Your Changes
 122  
 123  When you're happy with the code on your computer, you need to commit the changes to git:
 124  
 125      $ git commit -a -m "Here is a commit message on what I changed in this commit"
 126  
 127  <a name="update-master"></a>
 128  ### Update Master
 129  
 130  It's pretty likely that other changes to master have happened while you were working. Go get them:
 131  
 132      $ git checkout master
 133      $ git pull
 134  
 135  Now reapply your patch on top of the latest changes:
 136  
 137      $ git checkout my_new_branch
 138      $ git rebase master
 139  
 140  No conflicts? Change still seems reasonable to you? Then move on.
 141  
 142  <a name="fork"></a>
 143  ### Fork
 144  
 145  Navigate to the Bonfire [GitHub repository](https://github.com/ci-bonfire/Bonfire) and press "Fork" in the upper right hand corner.
 146  
 147  Add the new remote to your local repository on your local machine:
 148  
 149      $ git remote add mine git@github.com:<your user name>/Bonfire.git
 150  
 151  Push to your remote:
 152  
 153      $ git push mine my_new_branch
 154  
 155  <a name="issue-pull-request"></a>
 156  ### Issue a Pull Request
 157  
 158  Navigate to the Bonfire repository you just pushed to (e.g. https://github.com/your-user-name/Bonfire) and press "Pull Request" in the upper right hand corner.
 159  
 160  Write your branch name in branch field (is filled with master by default) and press "Update Commit Range"
 161  
 162  Ensure the changesets you introduced are included in the "Commits" tab and that the "Files Changed" incorporate all of your changes.
 163  
 164  Fill in some details about your potential patch including a meaningful title. When finished, press "Send pull request." Bonfire Core will be notified about your submission.
 165  
 166  <a name="iterate"></a>
 167  ### Iterate as Necessary
 168  
 169  It's entirely possible that the feedback you get will suggest changes. Don't get discouraged: the whole point of contributing to an active open source project is to tap into community knowledge. If people are encouraging you to tweak your code, then it's worth making the tweaks and resubmitting.
 170  
 171  And then...think about your next contribution!
