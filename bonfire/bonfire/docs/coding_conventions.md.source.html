   1  ## Coding Conventions
   2  
   3  ### File Format
   4  All files should be in the UTF-8 format. Line endings should be set to Unix-style line endings (LF). It is possible to have git auto-correct the line-endings for you. Please see [GitHub’s Help Pages](http://help.github.com/dealing-with-lineendings/) for more information.
   5  
   6  ### Tabs
   7  Spaces, not tabs, are preferred.
   8  
   9  ### Code Indentation
  10  Use Allman-style indenting. With the exception of class declarations, braces are always placed on a line by themselves and indented at the same level as the parent that ‘owns’ them.
  11  
  12      class Class_name {
  13  
  14          public function method_name()
  15          {
  16              foreach ($array as $item)
  17              {
  18                  // Do something
  19              }
  20          }
  21  
  22          //----------------------------------------------------------------------
  23      }
  24  
  25  ### PHP Closing Tag
  26  The PHP closing tag for the file should be _omitted_. This protects files from echoing data before they need to.
  27  
  28  ### Visibility
  29  All class variables and methods must have visibility declared.
  30  
  31      class Class_name {
  32  
  33          public     $property;
  34          protected  $other_property;
  35          private    $ci;
  36  
  37          //--------------------------------------------------------------------
  38  
  39          public function method_name()
  40          {
  41  
  42          }
  43  
  44          //--------------------------------------------------------------------
  45      }
  46  
  47  ### Class and Method Naming
  48  Class names should always have their first letter uppercase.
  49  
  50  Use the PHP 5 *__construct()* function for constructors unless absolutely necessary.
  51  
  52  Multiple words should be separated with an underscore, and not CamelCased.
  53  
  54  All other class methods should be entirely lowercased and named to clearly indicate their function.
  55  
  56  ### Variable Naming
  57  All variable names should describe what they are without being overly verbose.
  58  
  59  They should only contain letters, numbers and underscores (no CamelCase).
  60  
  61  Avoid short variable names whenever possible, and opt for a more descriptive name.
  62  
  63  ### Readable Code
  64  You should strive to make your code as easy to read as possible. This makes it easier for other developers to get familiar with your code.
  65  
  66  #### Avoid Nesting
  67  Avoid deep nesting of logic checks where possible. This only serves to make the code confusing down the road. Instead, check for an error condition and return, or similar function, as early as possible.
  68  
  69  GOOD:
  70  
  71      public function method_name()
  72      {
  73          if ( ! isset($var))
  74          {
  75              return false;
  76          }
  77  
  78          if ($other_var !== TRUE)
  79          {
  80              return false;
  81          }
  82  
  83          foreach ($this as $that)
  84          {
  85              // Code here...
  86          }
  87      }
  88  
  89      //--------------------------------------------------------------------
  90  
  91  
  92  BAD:
  93  
  94      public function method_name()
  95      {
  96          if (isset($var))
  97          {
  98              if ($other_var === TRUE)
  99              {
 100                  foreach ($this as $that)
 101                  {
 102                      if ($that == $something)
 103                      {
 104                          // Code here...
 105                      }
 106                  }
 107              }
 108          }
 109      }
 110  
 111      //--------------------------------------------------------------------
 112  
 113  #### Comment as Needed
 114  Comments should be used as frequently as needed, but not overdone. If your variable names and method names are descriptive then your code will be understandable without a lot of comments.
 115  
 116  This doesn’t mean don’t comment at all. That’s even worse.
 117  
 118  Use comments to
 119  
 120  - Describe why you are doing something (not how - that’s what the code is for)
 121  - Separate the logic into self-contained chunks to make finding your place easier.
 122  - As a reminder of what something is, or where it came from (if it is handled in another class).
