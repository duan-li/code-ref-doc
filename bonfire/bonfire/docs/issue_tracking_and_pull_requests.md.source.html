   1  # Issue Guidelines
   2  
   3  Please read this document before submitting an issue or pull request the CI-Bonfire project.
   4  It really helps us out and lets us know that you respect our time.
   5  In return, we will show the same respect in addressing your issue.
   6  
   7  If you think this guide can be improved, please let us know.
   8  
   9  ## Types of issues<a name="types-of-issues"></a>
  10  
  11  The GitHub issue tracker should only be used for one of the following:
  12  
  13  + **Bugs** &mdash; when a feature of the project has been _identified as
  14    broken_.
  15  + **Feature requests** &mdash; when you ask for a _new feature_ to be added to a
  16    Bonfire.
  17  + **Contribution enquiries** &mdash; when you want to discuss whether a _new
  18    feature_ or _change_ would be accepted in Bonfire before you begin
  19  development work on it.
  20  
  21  These are some things that don't belong in the issue tracker:
  22  
  23  + **Please avoid personal support requests.** We try to provide the best support
  24  we can, while maintaining a free project that helps you code faster.  We have forums,
  25  user guides, API doc's, CodeIgniter has great user doc's and a execellent forum.
  26  
  27  + **Please avoid derailing issues.** Keep the discussion on topic and respect the
  28  opinions of others.
  29  
  30  ## Bugs<a name="bugs"></a>
  31  
  32  A bug is a _demonstrable problem_ that is caused by the code in the
  33  repository.
  34  
  35  If you've come across a problem with the code and you're letting us know about
  36  it, _thank you_. We appreciate your time and the effort you're making to help
  37  improve the code for everyone else!
  38  
  39  Please read the following guidelines for reporting bugs:
  40  
  41  1. **Use the GitHub issue search** &mdash; check if the issue has already been
  42     reported. If it has been, please comment on the existing issue.
  43  
  44  2. **Check if the issue has been fixed** &mdash; the latest `master` or
  45     development branch may already contain a fix.
  46  
  47  3. **Isolate the demonstrable problem** &mdash; make sure that the code in the
  48     project's repository is _definitely_ responsible for the issue. Create a
  49  [reduced test case](http://css-tricks.com/6263-reduced-test-cases/) - an
  50  extremely simple and immediately viewable example of the issue.
  51  
  52  4. **Include a live example** &mdash; provide a link to your reduced test case
  53     when appropriate (e.g. if the issue is related to frond-end technologies).
  54  Please use [jsFiddle](http://jsfiddle.net) to host examples.
  55  
  56  Please try to be as detailed as possible in your report too. What is your
  57  environment? What steps will reproduce the issue? What browser(s) and OS
  58  experience the problem? What would you expect to be the outcome? All these
  59  details will help me and others to assess and fix any potential bugs.
  60  
  61  ### Example of a good bug report:
  62  
  63  > Short and descriptive title
  64  >
  65  > A summary of the issue and the browser/OS environment in which it occurs. If
  66  > suitable, include the steps required to reproduce the bug.
  67  >
  68  > 1. This is the first step
  69  > 2. This is the second step
  70  > 3. Further steps, etc.
  71  >
  72  > `<url>` - a link to the reduced test case
  73  >
  74  > Any other information you want to share that is relevant to the issue being
  75  > reported. This might include the lines of code that you have identified as
  76  > causing the bug, and potential solutions (and your opinions on their
  77  > merits).
  78  
  79  A good bug report shouldn't leave us needing to chase you up to get further
  80  information that is required to assess or fix the bug.
  81  
  82  ## Feature requests<a name="feature-requests"></a>
  83  
  84  Feature requests are welcome! Please provide links to examples or articles that
  85  help to illustrate the specifics of a feature you're requesting. The more
  86  detail, the better. It will help us to decide whether the feature is something we
  87  agree should become part of the project.
  88  
  89  ## Contribution enquiries<a name="enquiries"></a>
  90  
  91  Contribution enquiries should take place before any significant pull request,
  92  otherwise you risk spending a lot of time working on something that we might not
  93  want to pull into the repository.
  94  
  95  In this regard, some contribution enquires may be feature requests that you
  96  would like to have a go at implementing yourself if they are wanted. Other
  97  enquiries might revolve around refactoring code or porting a project to
  98  different languages.
  99  
 100  
 101  ## Pull requests<a name="pull-requests"></a>
 102  
 103  Good pull requests - patches, improvements, new features - are a fantastic
 104  help.
 105  
 106  If you've spotted any small, obvious errors and want to help out by patching it,
 107  that will be much appreciated.
 108  
 109  If your contribution involves a significant amount of work or substantial
 110  changes to any part of the project, please open a "contribution enquiry" issue
 111  first to check that the work is wanted or matches the goals of the project.
 112  
 113  All pull requests should remain focused in scope and avoid containing unrelated
 114  commits.
 115  
 116  Please follow this process; it's the best way to get your work merged into the
 117  project:
 118  
 119  1. [Fork](http://help.github.com/fork-a-repo/) the project.
 120  2. Clone your fork ( `git clone
 121     git@github.com:<your-username>/<repo-name>git`).
 122  3. Add an `upstream` remote (`git remote add upstream
 123     git://github.com/<upsteam-owner>/<repo-name>.git`).
 124  4. Get the latest changes from upstream (e.g. `git pull upstream
 125     <dev-branch>`).
 126  5. Create a new topic branch to contain your feature, change, or fix ( `git
 127     checkout -b <topic-branch-name>` ).
 128  6. Make sure that your changes adhere to the current [coding conventions](coding_conventions) used
 129     throughout the project - indentation, accurate comments, etc.
 130  7. Commit your changes in logical chunks. Please adhere to these [git commit
 131     message guidelines](http://tbaggery.com/2008/04/19/a-note-about-git-commit-messages.html)
 132     or your pull request is unlikely be merged into the main project.
 133  8. Push the branch up to your fork ( `git push origin <topic-branch-name>` ).
 134  9. [Open a Pull Request](http://help.github.com/send-pull-requests/) with a
 135     clear title and description. Please mention which browsers you tested in.
