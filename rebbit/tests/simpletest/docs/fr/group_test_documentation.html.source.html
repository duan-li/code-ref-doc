  33  <h1>Documentation sur le groupement des tests</h1>
  34          This page...
  35          <ul>
  36  <li>
  37              Plusieurs approches pour <a href="#group">grouper des tests</a> ensemble.
  38          </li>
  39  <li>
  40              Combiner des groupes des tests dans des
  41              <a href="#plus-haut">groupes plus grands</a>.
  42          </li>
  43  </ul>
  44  <div class="content">
  45          <h2>
  46  <a class="target" name="grouper"></a>Grouper des tests</h2>
  47              <p>
  48                  Il existe beaucoup de moyens pour grouper des tests dans des suites de tests.
  49                  Le premier d'entre eux est tout simplement ajouter tous les scénarios de test
  50                  au fur et à mesure d'un unique fichier...
  51  <pre>
  52  <strong>&lt;?php
  53  require_once(dirname(__FILE__) . '/simpletest/autorun.php');
  54  require_once(dirname(__FILE__) . '/../classes/io.php');
  55  
  56  class FileTester extends UnitTestCase {
  57      ...
  58  }
  59  
  60  class SocketTester extends UnitTestCase {
  61      ...
  62  }
  63  ?&gt;</strong>
  64  </pre>
  65                  Autant de scénarios que nécessaires peuvent être
  66                  mis dans cet unique fichier. Ils doivent contenir
  67                  tout le code nécessaire, entre autres la bibliothèque testée,
  68                  mais aucune des bibliothèques de SimpleTest.
  69              </p>
  70              <p>
  71                  Occasionnellement des sous-classes spéciales sont créés pour
  72                  ajouter des méthodes nécessaires à certains tests spécifiques
  73                  au sein de l'application.
  74                  Ces nouvelles classes de base sont ensuite utilisées
  75                  à la place de <span class="new_code">UnitTestCase</span>
  76                  ou de <span class="new_code">WebTestCase</span>.
  77                  Bien sûr vous ne souhaitez pas les lancer en tant que
  78                  scénario de tests : il suffit alors de les identifier
  79                  comme "abstraites"...
  80  <pre>
  81  <strong>abstract</strong> class MyFileTestCase extends UnitTestCase {
  82      ...
  83  }
  84  
  85  class FileTester extends MyFileTestCase { ... }
  86  
  87  class SocketTester extends UnitTestCase { ... }
  88  </pre>
  89                  La classe <span class="new_code">FileTester</span> ne contient aucun test véritable,
  90                  il s'agit d'une classe de base pour d'autres scénarios de test.
  91              </p>
  92              <p>
  93                  Nous appelons ce fichier <em>file_test.php</em>.
  94                  Pour l'instant les scénarios de tests sont simplement groupés dans le même fichier.
  95                  Nous pouvons mettre en place des structures plus importantes
  96                  en incluant d'autres fichiers de tests.
  97  <pre>
  98  &lt;?php
  99  require_once('simpletest/autorun.php');
 100  require_once('file_test.php');
 101  ?&gt;
 102  </pre>
 103                  Ceci fontionnera, tout en créant une hiérarchie tout à fait plate.
 104                  A la place, nous créons un fichier de suite de tests.
 105                  Notre suite des tests de premier niveau devient...
 106  <pre>
 107  &lt;?php
 108  require_once('simpletest/autorun.php');
 109  
 110  class AllFileTests extends TestSuite {
 111      function __construct() {
 112          parent::__construct();
 113          <strong>$this-&gt;addFile('file_test.php');</strong>
 114      }
 115  }
 116  ?&gt;
 117  </pre>
 118                  Voici ce qui arrive : la class <span class="new_code">TestSuite</span>
 119                  effecturera le <span class="new_code">require_once()</span> pour nous.
 120                  Ensuite elle vérifie si de nouvelles classes de test
 121                  ont été créées par ce nouveau fichier et les inclut
 122                  automatiquement dans la suite de tests.
 123                  Cette méthode nous donne un maximum de contrôle
 124                  tout comme le ferait des ajouts manuels de fichiers de tests
 125                  au fur et à mesure où notre suite grandit.
 126              </p>
 127              <p>
 128                  Si c'est trop de boulot pour vos petits doigts et qu'en plus
 129                  vous avez envie de grouper vos suites de tests par répertoire
 130                  ou par un tag dans le nom des fichiers, alors il y a un moyen
 131                  encore plus automatique... 
 132  <pre>
 133  &lt;?php
 134  require_once('simpletest/autorun.php');
 135  
 136  class AllFileTests extends TestSuite {
 137      function __construct() {
 138          parent::__construct();
 139          $this-&gt;collect(dirname(__FILE__) . '/unit',
 140                         new SimplePatternCollector('/_test.php/'));
 141      }
 142  }
 143  ?&gt;
 144  </pre>
 145                  Cette fonctionnalités va scanner un répertoire appelé "unit",
 146                  y détecter tous les fichiers finissant par "_test.php"
 147                  et les charger.
 148                  Vous n'avez pas besoin d'utiliser <span class="new_code">SimplePatternCollector</span> pour
 149                  filtrer en fonction d'un motif dans le nom de fichier,
 150                  mais c'est son usage le plus courant. 
 151              </p>
 152              <p>
 153                  Ce morceau de code est très courant.
 154                  Désormais la seule chose qu'il vous reste à faire, c'est de
 155                  déposer un fichier avec des scénarios de tests dans ce répertoire
 156                  et il sera lancé directement par le script de la suite de tests. 
 157              </p>
 158              <p>
 159                  Juste un bémol : vous ne pouvez pas contrôler l'ordre de lancement
 160                  des tests.
 161                  Si vous souhaitez voir des composants de bas niveau renvoyer leurs erreurs
 162                  dès le début de la suite de tests - en particulier pour se facilier le travail
 163                  de diagnostic - alors vous devriez plutôt passer par <span class="new_code">addFile()</span>
 164                  pour ces cas spécifiques.
 165                  Les scénarios de tests ne sont chargés qu'une fois, pas d'inquiétude donc
 166                  lors du scan de votre répertoire avec tous les tests.
 167              </p>
 168              <p>
 169                  Les tests chargés avec la méthode <span class="new_code">addFile</span> ont certaines propriétés
 170                  qui peuvent s'avérer intéressantes.
 171                  Elle vous assure que le constructeur est lancé avant la première méthode
 172                  de test et que le destructeur est lancé après la dernière méthode de test.
 173                  Cela vous permet de placer une initialisation (setUp et tearDown) globale
 174                  au sein de ce destructeur et desctructeur, comme dans n'importe
 175                  quelle classe. 
 176              </p>
 177          
 178          <h2>
 179  <a class="target" name="plus-haut"></a>Groupements de plus haut niveau</h2>
 180              <p>
 181                  La technique ci-dessus place tous les scénarios de test
 182                  dans un unique et grand groupe.
 183                  Sauf que pour des projets plus conséquents,
 184                  ce n'est probablement pas assez souple;
 185                  vous voudriez peut-être grouper les tests tout à fait différemment.
 186              </p>
 187  <p>
 188                  Tout ce que nous avons décrit avec des scripts de tests 
 189                  s'applique pareillement avec des <span class="new_code">TestSuite</span>s...
 190  <pre>
 191  &lt;?php
 192  require_once('simpletest/autorun.php');
 193  <strong>
 194  class BigTestSuite extends TestSuite {
 195      function __construct() {
 196          parent::__construct();
 197          $this-&gt;addFile('file_tests.php');
 198      }
 199  }</strong>
 200  ?&gt;
 201  </pre>
 202                  Cette opération additionne nos scénarios de tests et une unique suite
 203                  sous la première.
 204                  Quand un test échoue, nous voyons le fil d'ariane avec l'enchainement.
 205                  Nous pouvons même mélanger groupes et tests librement en prenant
 206                  quand même soin d'éviter les inclusions en boucle.
 207  <pre>
 208  &lt;?php
 209  require_once('simpletest/autorun.php');
 210  
 211  class BigTestSuite extends TestSuite {
 212      function __construct() {
 213          parent::__construct();
 214          $this-&gt;addFile('file_tests.php');
 215          <strong>$this-&gt;addFile('some_other_test.php');</strong>
 216      }
 217  }
 218  ?&gt;
 219  </pre>
 220                  Petite précision, en cas de double inclusion, seule la première instance
 221                  sera lancée. 
 222              </p>
