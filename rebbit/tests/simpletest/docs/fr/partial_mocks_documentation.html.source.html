  33  <h1>Documentation sur les objets fantaisie partiels</h1>
  34          This page...
  35          <ul>
  36  <li>
  37              <a href="#injection">Le problème de l'injection d'un objet fantaisie</a>.
  38          </li>

  41          </li>

  49  <div class="content">
  50          
  51              <p>
  52                  Un objet fantaisie partiel n'est ni plus ni moins
  53                  qu'un modèle de conception pour soulager un problème spécifique
  54                  du test avec des objets fantaisie, celui de placer
  55                  des objets fantaisie dans des coins serrés.
  56                  Il s'agit d'un outil assez limité et peut-être même
  57                  une idée pas si bonne que ça. Elle est incluse dans SimpleTest
  58                  pour la simple raison que je l'ai trouvée utile
  59                  à plus d'une occasion et qu'elle m'a épargnée
  60                  pas mal de travail dans ces moments-là.
  61              </p>
  62          
  63          <h2>
  64  <a class="target" name="injection"></a>Le problème de l'injection dans un objet fantaisie</h2>
  65              <p>
  66                  Quand un objet en utilise un autre il est très simple
  67                  d'y faire circuler une version fantaisie déjà prête
  68                  avec ses attentes. Les choses deviennent un peu plus délicates
  69                  si un objet en crée un autre et que le créateur est celui
  70                  que l'on souhaite tester. Cela revient à dire que l'objet
  71                  créé devrait être une fantaisie, mais nous pouvons
  72                  difficilement dire à notre classe sous test de créer
  73                  un objet fantaisie plutôt qu'un "vrai" objet.
  74                  La classe testée ne sait même pas qu'elle travaille dans un environnement de test.
  75              </p>
  76              <p>
  77                  Par exemple, supposons que nous sommes en train
  78                  de construire un client telnet et qu'il a besoin
  79                  de créer une socket réseau pour envoyer ses messages.
  80                  La méthode de connexion pourrait ressemble à quelque chose comme...
  81  <pre>
  82  <strong>&lt;?php
  83  require_once('socket.php');
  84  
  85  class Telnet {
  86      ...
  87      function connect($ip, $port, $username, $password) {
  88          $socket = new Socket($ip, $port);
  89          $socket-&gt;read( ... );
  90          ...
  91      }
  92  }
  93  ?&gt;</strong>
  94  </pre>
  95                  Nous voudrions vraiment avoir une version fantaisie
  96                  de l'objet socket, que pouvons nous faire ?
  97              </p>
  98              <p>
  99                  La première solution est de passer la socket en
 100                  tant que paramètre, ce qui force la création
 101                  au niveau inférieur. Charger le client de cette tâche
 102                  est effectivement une bonne approche si c'est possible
 103                  et devrait conduire à un remaniement -- de la création
 104                  à partir de l'action. En fait, c'est là une des manières
 105                  avec lesquels tester en s'appuyant sur des objets fantaisie
 106                  vous force à coder des solutions plus resserrées sur leur objectif.
 107                  Ils améliorent votre programmation.
 108              </p>
 109              <p>
 110                  Voici ce que ça devrait être...
 111  <pre>
 112  &lt;?php
 113  require_once('socket.php');
 114  
 115  class Telnet {
 116      ...
 117      <strong>function connect($socket, $username, $password) {
 118          $socket-&gt;read( ... );
 119          ...
 120      }</strong>
 121  }
 122  ?&gt;
 123  </pre>
 124                  Sous-entendu, votre code de test est typique d'un cas
 125                  de test avec un objet fantaisie.
 126  <pre>
 127  class TelnetTest extends UnitTestCase {
 128      ...
 129      function testConnection() {<strong>
 130          $socket = new MockSocket();
 131          ...
 132          $telnet = new Telnet();
 133          $telnet-&gt;connect($socket, 'Me', 'Secret');
 134          ...</strong>
 135      }
 136  }
 137  </pre>
 138                  C'est assez évident que vous ne pouvez descendre que d'un niveau.
 139                  Vous ne voudriez pas que votre application de haut niveau
 140                  crée tous les fichiers de bas niveau, sockets et autres connexions
 141                  à la base de données dont elle aurait besoin.
 142                  Elle ne connaîtrait pas les paramètres du constructeur de toute façon.
 143              </p>
 144              <p>
 145                  La solution suivante est de passer l'objet créé sous la forme
 146                  d'un paramètre optionnel...
 147  <pre>
 148  &lt;?php
 149  require_once('socket.php');
 150  
 151  class Telnet {
 152      ...<strong>
 153      function connect($ip, $port, $username, $password, $socket = false) {
 154          if (! $socket) {
 155              $socket = new Socket($ip, $port);
 156          }
 157          $socket-&gt;read( ... );</strong>
 158          ...
 159          return $socket;
 160      }
 161  }
 162  ?&gt;
 163  </pre>
 164                  Pour une solution rapide, c'est généralement suffisant.
 165                  Ensuite le test est très similaire : comme si le paramètre
 166                  était transmis formellement...
 167  <pre>
 168  class TelnetTest extends UnitTestCase {
 169      ...
 170      function testConnection() {<strong>
 171          $socket = new MockSocket();
 172          ...
 173          $telnet = new Telnet();
 174          $telnet-&gt;connect('127.0.0.1', 21, 'Me', 'Secret', $socket);
 175          ...</strong>
 176      }
 177  }
 178  </pre>
 179                  Le problème de cette approche tient dans son manque de netteté.
 180                  Il y a du code de test dans la classe principale et aussi
 181                  des paramètres transmis dans le scénario de test
 182                  qui ne sont jamais utilisés. Il s'agit là d'une approche
 183                  rapide et sale, mais qui ne reste pas moins efficace
 184                  dans la plupart des situations.
 185              </p>
 186              <p>
 187                  Une autre solution encore est de laisser un objet fabrique
 188                  s'occuper de la création...
 189  <pre>
 190  &lt;?php
 191  require_once('socket.php');
 192  
 193  class Telnet {<strong>
 194     function Telnet($network) {
 195          $this-&gt;_network = $network;
 196      }</strong>
 197      ...
 198      function connect($ip, $port, $username, $password) {<strong>
 199          $socket = $this-&gt;_network-&gt;createSocket($ip, $port);
 200          $socket-&gt;read( ... );</strong>
 201          ...
 202          return $socket;
 203      }
 204  }
 205  ?&gt;
 206  </pre>
 207                  Il s'agit là probablement de la réponse la plus travaillée
 208                  étant donné que la création est maintenant située
 209                  dans une petite classe spécialisée. La fabrique réseau
 210                  peut être testée séparément et utilisée en tant que fantaisie
 211                  quand nous testons la classe telnet...
 212  <pre>
 213  class TelnetTest extends UnitTestCase {
 214      ...
 215      function testConnection() {<strong>
 216          $socket = new MockSocket();
 217          ...
 218          $network = new MockNetwork();
 219          $network-&gt;returnsByReference('createSocket', $socket);
 220          $telnet = new Telnet($network);
 221          $telnet-&gt;connect('127.0.0.1', 21, 'Me', 'Secret');</strong>
 222      }
 223  }
 224  </pre>
 225                  Le problème reste que nous ajoutons beaucoup de classes
 226                  à la bibliothèque. Et aussi que nous utilisons beaucoup
 227                  de fabriques ce qui rend notre code un peu moins intuitif.
 228                  La solution la plus flexible, mais aussi la plus complexe.
 229              </p>
 230              <p>
 231                  Des techniques comme "l'Injection de Dépendance"
 232                  (ou "Dependency Injection") s'attelle au problème
 233                  de l'instanciation d'une classe avec beaucoup de paramètres.
 234                  Malheureusement la connaissance de ce patron de conception
 235                  n'est pas très répandue et si vous êtes en train d'essayer
 236                  de faire fonctionner du vieux code, ré-achitecturer toute
 237                  l'application n'est pas vraiment une option.
 238              </p>
 239              <p>
 240                  Peut-on trouver un juste milieu ?
 241              </p>
 242          
 243          <h2>
 244  <a class="target" name="creation"></a>Méthode fabrique protégée</h2>
 245              <p>
 246                  Il existe une technique pour palier à ce problème
 247                  sans créer de nouvelle classe dans l'application;
 248                  par contre elle induit la création d'une sous-classe au moment du test.
 249                  Premièrement nous déplaçons la création de la socket dans sa propre méthode...
 250  <pre>
 251  &lt;?php
 252  require_once('socket.php');
 253  
 254  class Telnet {
 255      ...
 256      function connect($ip, $port, $username, $password) {
 257          <strong>$socket = $this-&gt;createSocket($ip, $port);</strong>
 258          $socket-&gt;read( ... );
 259          ...
 260      }<strong>
 261  
 262      protected function createSocket($ip, $port) {
 263          return new Socket($ip, $port);
 264      }</strong>
 265  }
 266  ?&gt;
 267  </pre>
 268                  Une première étape plutôt précautionneuse même pour
 269                  du code legacy et intermélé.
 270                  Il s'agit là de la seule modification dans le code de l'application.
 271              </p>
 272              <p>
 273                  Pour le scénario de test, nous devons créer
 274                  une sous-classe de manière à intercepter la création de la socket...
 275  <pre>
 276  <strong>class TelnetTestVersion extends Telnet {
 277      var $mock;
 278  
 279      function TelnetTestVersion($mock) {
 280          $this-&gt;mock = $mock;
 281          $this-&gt;Telnet();
 282      }
 283  
 284      protected function createSocket() {
 285          return $this-&gt;mock;
 286      }
 287  }</strong>
 288  </pre>
 289                  Ici j'ai déplacé la fantaisie dans le constructeur,
 290                  mais un setter aurait fonctionné tout aussi bien.
 291                  Notez bien que la fantaisie est placée dans une variable
 292                  d'objet avant que le constructeur ne soit attaché.
 293                  C'est nécessaire dans le cas où le constructeur appelle 
 294                  <span class="new_code">connect()</span>.
 295                  Autrement il pourrait donner un valeur nulle à partir de
 296                  <span class="new_code">createSocket()</span>.
 297              </p>
 298              <p>
 299                  Après la réalisation de tout ce travail supplémentaire
 300                  le scénario de test est assez simple.
 301                  Nous avons juste besoin de tester notre nouvelle classe à la place...
 302  <pre>
 303  class TelnetTest extends UnitTestCase {
 304      ...
 305      function testConnection() {<strong>
 306          $socket = new MockSocket();
 307          ...
 308          $telnet = new TelnetTestVersion($socket);
 309          $telnet-&gt;connect('127.0.0.1', 21, 'Me', 'Secret');</strong>
 310      }
 311  }
 312  </pre>
 313                  Cette nouvelle classe est très simple bien sûr.
 314                  Elle ne fait qu'initier une valeur renvoyée, à la manière
 315                  d'une fantaisie. Ce serait pas mal non plus si elle pouvait
 316                  vérifier les paramètres entrants.
 317                  Exactement comme un objet fantaisie.
 318                  Il se pourrait bien que nous ayons à réaliser cette astuce régulièrement :
 319                  serait-il possible d'automatiser la création de cette sous-classe ?
 320              </p>

 322          <h2>
 323  <a class="target" name="partiel"></a>Un objet fantaisie partiel</h2>
 324              <p>
 325                  Bien sûr la réponse est "oui"
 326                  ou alors j'aurais arrêté d'écrire depuis quelques temps déjà !
 327                  Le test précédent a représenté beaucoup de travail,
 328                  mais nous pouvons générer la sous-classe en utilisant
 329                  une approche à celle des objets fantaisie.
 330              </p>
 331              <p>
 332                  Voici donc une version avec objet fantaisie partiel du test...
 333  <pre>
 334  <strong>Mock::generatePartial(
 335          'Telnet',
 336          'TelnetTestVersion',
 337          array('createSocket'));</strong>
 338  
 339  class TelnetTest extends UnitTestCase {
 340      ...
 341      function testConnection() {<strong>
 342          $socket = new MockSocket();
 343          ...
 344          $telnet = new TelnetTestVersion();
 345          $telnet-&gt;setReturnReference('createSocket', $socket);
 346          $telnet-&gt;Telnet();
 347          $telnet-&gt;connect('127.0.0.1', 21, 'Me', 'Secret');</strong>
 348      }
 349  }
 350  </pre>
 351                  La fantaisie partielle est une sous-classe de l'original
 352                  dont on aurait "remplacé" les méthodes sélectionnées
 353                  avec des versions de test. L'appel à <span class="new_code">generatePartial()</span>
 354                  nécessite trois paramètres : la classe à sous classer,
 355                  le nom de la nouvelle classe et une liste des méthodes à simuler.
 356              </p>
 357              <p>
 358                  Instancier les objets qui en résultent est plutôt délicat.
 359                  L'unique paramètre du constructeur d'un objet fantaisie partiel
 360                  est la référence du testeur unitaire.
 361                  Comme avec les objets fantaisie classiques c'est nécessaire
 362                  pour l'envoi des résultats de test en réponse à la vérification des attentes.
 363              </p>
 364              <p>
 365                  Une nouvelle fois le constructeur original n'est pas lancé.
 366                  Indispensable dans le cas où le constructeur aurait besoin
 367                  des méthodes fantaisie : elles n'ont pas encore été initiées !
 368                  Nous initions les valeurs retournées à cet instant et
 369                  ensuite lançons le constructeur avec ses paramètres normaux.
 370                  Cette construction en trois étapes de "new",
 371                  suivie par la mise en place des méthodes et ensuite
 372                  par la lancement du constructeur proprement dit est
 373                  ce qui distingue le code d'un objet fantaisie partiel.
 374              </p>
 375              <p>
 376                  A part pour leur construction, toutes ces méthodes
 377                  fantaisie ont les mêmes fonctionnalités que dans
 378                  le cas des objets fantaisie et toutes les méthodes
 379                  non fantaisie se comportent comme avant.
 380                  Nous pouvons mettre en place des attentes très facilement...
 381  <pre>
 382  class TelnetTest extends UnitTestCase {
 383      ...
 384      function testConnection() {
 385          $socket = new MockSocket();
 386          ...
 387          $telnet = new TelnetTestVersion();
 388          $telnet-&gt;setReturnReference('createSocket', $socket);
 389          <strong>$telnet-&gt;expectOnce('createSocket', array('127.0.0.1', 21));</strong>
 390          $telnet-&gt;Telnet();
 391          $telnet-&gt;connect('127.0.0.1', 21, 'Me', 'Secret');
 392      }
 393  }
 394  </pre>
 395                  Les objets fantaisie partiels ne sont pas très utilisés.
 396                  Je les considère comme transitoire.
 397                  Utile lors d'un remaniement, mais une fois que l'application
 398                  a eu toutes ses dépendances bien séparées alors
 399                  ils peuvent disparaître.
 400              </p>
 401          
 402          <h2>
 403  <a class="target" name="moins"></a>Tester moins qu'une classe</h2>
 404              <p>
 405                  Les méthodes issues d'un objet fantaisie n'ont pas
 406                  besoin d'être des méthodes fabrique, Il peut s'agir
 407                  de n'importe quelle sorte de méthode.
 408                  Ainsi les objets fantaisie partiels nous permettent
 409                  de prendre le contrôle de n'importe quelle partie d'une classe,
 410                  le constructeur excepté. Nous pourrions même aller jusqu'à
 411                  créer des fantaisies sur toutes les méthodes à part celle
 412                  que nous voulons effectivement tester.
 413              </p>
 414              <p>
 415                  Cette situation est assez hypothétique, étant donné
 416                  que je ne l'ai pas souvent essayée.
 417                  Je crains qu'en forçant la granularité d'un objet
 418                  on n'obtienne pas forcément un code de meilleur qualité.
 419                  Personnellement j'utilise les objets fantaisie partiels
 420                  comme moyen de passer outre la création ou alors
 421                  de temps en temps pour tester le modèle de conception TemplateMethod.
 422              </p>
 423              <p>
 424                  On en revient toujours aux standards de code de votre projet :
 425                  c'est à vous de trancher si vous autorisez ce mécanisme ou non.
 426              </p>
