  47          <h2>
  48  <a class="target" name="what"></a>Que sont les objets fantaisie ?</h2>
  49              <p>
  50                  Les objets fantaisie - ou "mock objects" en anglais -
  51                  ont deux rôles pendant un scénario de test : acteur et critique.
  52              </p>
  53              <p>
  54                  Le comportement d'acteur est celui de simuler
  55                  des objets difficiles à initialiser ou trop consommateurs
  56                  en temps pendant un test.
  57                  Le cas classique est celui de la connexion à une base de données.
  58                  Mettre sur pied une base de données de test au lancement
  59                  de chaque test ralentirait considérablement les tests
  60                  et en plus exigerait l'installation d'un moteur
  61                  de base de données ainsi que des données sur la machine de test.
  62                  Si nous pouvons simuler la connexion
  63                  et renvoyer des données à notre guise
  64                  alors non seulement nous gagnons en pragmatisme
  65                  sur les tests mais en sus nous pouvons nourrir
  66                  notre base avec des données falsifiées
  67                  et voir comment il répond. Nous pouvons
  68                  simuler une base de données en suspens ou
  69                  d'autres cas extrêmes sans avoir à créer
  70                  une véritable panne de base de données.
  71                  En d'autres termes nous pouvons gagner
  72                  en contrôle sur l'environnement de test.
  73              </p>
  74              <p>
  75                  Si les objets fantaisie ne se comportaient que comme
  76                  des acteurs alors on les connaîtrait sous le nom de
  77                  <a href="server_stubs_documentation.html">bouchons serveur</a>.
  78                  Il s'agissait originairement d'un patron de conception
  79                  identifié par Robert Binder (<a href="">Testing
  80                  object-oriented systems</a>: models, patterns, and tools,
  81                  Addison-Wesley) en 1999.
  82              </p>
  83              <p>
  84                  Un bouchon serveur est une simulation d'un objet ou d'un composant.
  85                  Il doit remplacer exactement un composant dans un système
  86                  en vue d'effectuer des tests ou un prototypage,
  87                  tout en restant ultra-léger.
  88                  Il permet aux tests de s'exécuter plus rapidement, ou 
  89                  si la classe simulée n'a pas encore été écrite,
  90                  de se lancer tout court.
  91              </p>
  92              <p>
  93                  Cependant non seulement les objets fantaisie jouent
  94                  un rôle (en fournissant à la demande les valeurs requises)
  95                  mais en plus ils sont aussi sensibles aux messages qui
  96                  leur sont envoyés (par le biais d'attentes).
  97                  En posant les paramètres attendus d'une méthode
  98                  ils agissent comme des gardiens :
  99                  un appel sur eux doit être réalisé correctement.
 100                  Si les attentes ne sont pas atteintes ils nous épargnent
 101                  l'effort de l'écriture d'une assertion de test avec
 102                  échec en réalisant cette tâche à notre place.
 103              </p>
 104              <p>
 105                  Dans le cas d'une connexion à une base de données
 106                  imaginaire ils peuvent tester si la requête, disons SQL,
 107                  a bien été formé par l'objet qui utilise cette connexion.
 108                  Mettez-les sur pied avec des attentes assez précises
 109                  et vous verrez que vous n'aurez presque plus d'assertion à écrire manuellement.
 110              </p>
 111          
 112          <h2>
 113  <a class="target" name="creation"></a>Créer des objets fantaisie</h2>
 114              <p>
 115                  Tout ce dont nous avons besoin est une classe existante ou une interface,
 116                  par exemple une connexion à la base de données qui ressemblerait à...
 117  <pre>
 118  <strong>class DatabaseConnection {
 119      function DatabaseConnection() { }
 120      function query($sql) { }
 121      function selectQuery($sql) { }
 122  }</strong>
 123  </pre>
 124                  Pour en créer sa version fantaisie nous devons juste
 125                  lancer le générateur...
 126  <pre>
 127  require_once('simpletest/autorun.php');
 128  require_once('database_connection.php');
 129  
 130  <strong>Mock::generate('DatabaseConnection');</strong>
 131  </pre>
 132                  Ce code génère une classe clone appelée <span class="new_code">MockDatabaseConnection</span>.
 133                  Cette nouvelle classe lui ressemble en tout point,
 134                  sauf qu'elle ne fait rien du tout.
 135              </p>
 136              <p>
 137                  Cette nouvelle classe est génératlement
 138                  une sous-classe de <span class="new_code">DatabaseConnection</span>.
 139                  Malheureusement, il n'y as aucun moyen de créer une version fantaisie
 140                  d'une classe avec une méthode <span class="new_code">final</span> sans avoir
 141                  une version fonctionnelle de cette méthode.
 142                  Ce n'est pas pas très satisfaisant.
 143                  Si la cible est une interface ou si les méthodes <span class="new_code">final</span>
 144                  existent dans la classe cible, alors une toute nouvelle classe
 145                  est créée, elle implémente juste les même interfaces.
 146                  Si vous essayer de faire passer cette classe à travers un indice de type
 147                  qui spécifie le véritable nom de l'ancienne classe, alors il échouera.
 148                  Du code qui forcerait un indice de type tout en utilisant
 149                  des méthodes <span class="new_code">final</span> ne pourrait probablement pas être
 150                  testé efficacement avec des objets fantaisie.
 151              </p>
 152              <p>
 153                  Si vous voulez voir le code généré, il suffit de faire un <span class="new_code">print</span>
 154                  de la sortie de <span class="new_code">Mock::generate()</span>.
 155                  VOici le code généré pour la classe <span class="new_code">DatabaseConnection</span>
 156                  à la place de son interface...
 157  <pre>
 158  class MockDatabaseConnection extends DatabaseConnection {
 159      public $mock;
 160      protected $mocked_methods = array('databaseconnection', 'query', 'selectquery');
 161  
 162      function MockDatabaseConnection() {
 163          $this-&gt;mock = new SimpleMock();
 164          $this-&gt;mock-&gt;disableExpectationNameChecks();
 165      }
 166      ...
 167      function DatabaseConnection() {
 168          $args = func_get_args();
 169          $result = &amp;$this-&gt;mock-&gt;invoke("DatabaseConnection", $args);
 170          return $result;
 171      }
 172      function query($sql) {
 173          $args = func_get_args();
 174          $result = &amp;$this-&gt;mock-&gt;invoke("query", $args);
 175          return $result;
 176      }
 177      function selectQuery($sql) {
 178          $args = func_get_args();
 179          $result = &amp;$this-&gt;mock-&gt;invoke("selectQuery", $args);
 180          return $result;
 181      }
 182  }
 183  </pre>
 184                  Votre sortie dépendra quelque peu de votre version précise de SimpleTest.
 185              </p>
 186              <p>
 187                  En plus des méthodes d'origine de la classe, vous en verrez d'autres
 188                  pour faciliter les tests.
 189                  Nous y reviendrons.
 190              </p>
 191              <p>
 192                  Nous pouvons désormais créer des instances de
 193                  cette nouvelle classe à l'intérieur même de notre scénario de test...
 194  <pre>
 195  require_once('simpletest/autorun.php');
 196  require_once('database_connection.php');
 197  
 198  Mock::generate('DatabaseConnection');
 199  
 200  class MyTestCase extends UnitTestCase {
 201  
 202      function testSomething() {
 203          <strong>$connection = new MockDatabaseConnection();</strong>
 204      }
 205  }
 206  </pre>
 207                  La version fantaisie contient toutles méthodes de l'originale.
 208                  En outre, tous les indices de type seront préservés.
 209                  Dison que nos méthodes de requête attend un objet <span class="new_code">Query</span>...
 210  <pre>
 211  <strong>class DatabaseConnection {
 212      function DatabaseConnection() { }
 213      function query(Query $query) { }
 214      function selectQuery(Query $query) { }
 215  }</strong>
 216  </pre>
 217                  Si nous lui passons le mauvais type d'objet
 218                  ou même pire un non-objet...
 219  <pre>
 220  class MyTestCase extends UnitTestCase {
 221  
 222      function testSomething() {
 223          $connection = new MockDatabaseConnection();
 224          $connection-&gt;query('insert into accounts () values ()');
 225      }
 226  }
 227  </pre>
 228                  ...alors le code renverra une violation de typage,
 229                  exactement comme on aurait pû s'y attendre
 230                  avec la classe d'origine.
 231              </p>
 232              <p>
 233                  Si la version fantaisie implémente bien toutes les méthodes
 234                  de l'originale, elle renvoit aussi systématiquement <span class="new_code">null</span>.
 235                  Et comme toutes les méthodes qui renvoient toujours <span class="new_code">null</span>
 236                  ne sont pas très utiles, nous avons besoin de leur faire dire
 237                  autre chose...
 238              </p>
 239          
 240          <h2>
 241  <a class="target" name="bouchon"></a>Objets fantaisie en action</h2>
 242              <p>
 243                  Changer le <span class="new_code">null</span> renvoyé par la méthode fantaisie
 244                  en autre chose est assez facile...
 245  <pre>
 246  <strong>$connection-&gt;returns('query', 37)</strong>
 247  </pre>
 248                  Désormais à chaque appel de <span class="new_code">$connection-&gt;query()</span>
 249                  nous obtenons un résultat de 37.
 250                  Il n'y a rien de particulier à cette valeur de 37.
 251                  Cette valeur de retour peut être aussi compliqué que nécessaire.
 252              </p>
 253              <p>
 254                  Ici les paramètres ne sont pas significatifs,
 255                  nous aurons systématiquement la même valeur en retour
 256                  une fois initialisés de la sorte.
 257                  Cela pourrait ne pas ressembler à une copie convaincante
 258                  de notre connexion à la base de données, mais pour
 259                  la demi-douzaine de lignes de code de notre méthode de test
 260                  c'est généralement largement assez.
 261              </p>
 262              <p>
 263                  Sauf que les choses ne sont pas toujours si simples.
 264                  Les itérateurs sont un problème récurrent, si par exemple
 265                  renvoyer systématiquement la même valeur entraine
 266                  une boucle infinie dans l'objet testé.
 267                  Pour ces cas-là, nous avons besoin d'une séquence de valeurs.
 268                  Supposons que nous avons un itérateur simple qui ressemble à...
 269  <pre>
 270  class Iterator {
 271      function Iterator() { }
 272      function next() { }
 273  }
 274  </pre>
 275                  Il s'agit là de l'itérateur le plus basique que nous puissions imaginer.
 276                  Supponsons que cet itérateur ne renvoit que du texte
 277                  jusqu'à ce qu'il atteigne la fin et qu'il renvoie alors un false,
 278                  nous pouvons le simuler avec...
 279  <pre>
 280  Mock::generate('Iterator');
 281  
 282  class IteratorTest extends UnitTestCase() {
 283  
 284      function testASequence() {<strong>
 285          $iterator = new MockIterator();
 286          $iterator-&gt;returns('next', false);
 287          $iterator-&gt;returnsAt(0, 'next', 'First string');
 288          $iterator-&gt;returnsAt(1, 'next', 'Second string');</strong>
 289          ...
 290      }
 291  }
 292  </pre>
 293                  Quand <span class="new_code">next()</span> est appelé par le <span class="new_code">MockIterator</span>
 294                  il commencera par renvoyer "First string",
 295                  au deuxième passage "Second string" sera renvoyé
 296                  et sur n'importe quel autre appel <span class="new_code">false</span> sera renvoyé
 297                  à son tour.
 298                  Les valeurs renvoyées les unes après les autres auront la priorité
 299                  sur la valeur constante.
 300                  Cette dernière est la valeur par défaut en quelque sorte.
 301              </p>
 302              <p>
 303                  Une autre situation délicate serait une opération
 304                  <span class="new_code">get()</span> surchargée.
 305                  Un exemple serait un conteneur d'information avec des pairs clef/valeur.
 306                  Nous partons cette fois d'une classe de configuration telle...
 307  <pre>
 308  class Configuration {
 309      function Configuration() { ... }
 310      function get($key) { ... }
 311  }
 312  </pre>
 313                  C'est une situation courante pour utiliser des objets fantaisie,
 314                  étant donné que la véritable configuration sera différente
 315                  d'une machine à l'autre et parfois même d'un test à l'autre.
 316                  Cependant un problème apparaît quand toutes les données passent
 317                  par la méthode <span class="new_code">get()</span> et que nous souhaitons
 318                  quand même des résultats différents pour des clefs différentes.
 319                  Par chance les objets fantaisie ont un système de filtre...
 320  <pre>
 321  <strong>$config = &amp;new MockConfiguration();
 322  $config-&gt;returns('get', 'primary', array('db_host'));
 323  $config-&gt;returns('get', 'admin', array('db_user'));
 324  $config-&gt;returns('get', 'secret', array('db_password'));</strong>
 325  </pre>
 326                  Le dernier paramètre est une liste d'arguements
 327                  pour vérifier une correspondance.
 328                  Dans ce cas, nous essayons de faire correspondre un argument
 329                  qui se trouve être la clef de recherche.
 330                  Désormais quand l'objet fantaisie voit
 331                  sa méthode <span class="new_code">get()</span> invoquée...
 332  <pre>
 333  $config-&gt;get('db_user')
 334  </pre>
 335                  ...il renverra "admin".
 336                  Il trouve cette valeur en essayant de faire correspondre
 337                  l'argument reçu à ceux de ses propres listes : dès qu'une
 338                  correspondance complète est trouvé, il s'arrête.
 339              </p>
 340              <p>
 341                  Vous pouvez préciser un argument par défaut via...
 342  <pre><strong>
 343  $config-&gt;returns('get', false, array('*'));</strong>
 344  </pre>
 345                  Ce n'est pas la même chose que de définir la valeur renvoyée
 346                  sans aucun arguement...
 347  <pre><strong>
 348  $config-&gt;returns('get', false);</strong>
 349  </pre>
 350                  Dans le premier cas, il acceptera n'importe quel argument,
 351                  mais il en faut au moins un.
 352                  Dans le deuxième cas, n'importe quel nombre d'arguments
 353                  fera l'affaire and il agit comme un attrape-tout après
 354                  toutes les autres vérifications.
 355                  Notez que si - dans le premier cas - nous ajoutons
 356                  d'autres options avec paramètre unique après le joker,
 357                  alors elle seront ignorés puisque le joker fera
 358                  la première correspondance.
 359                  Avec des listes complexes de paramètres, l'ordre devient
 360                  important au risque de voir la correspondance souhaitée
 361                  masqué par un joker apparu plus tôt.
 362                  Déclarez les plus spécifiques d'abord si vous n'êtes pas sûr.
 363              </p>
 364              <p>
 365                  Il y a des moments où vous souhaitez qu'une référence
 366                  bien spécifique soit servie par l'objet fantaisie plutôt
 367                  qu'une copie.
 368                  C'est plutôt rare pour dire le moins, mais vous pourriez 
 369                  être en train de simuler un conteneur qui détiendrait
 370                  des primitives, tels des chaînes de caractères.
 371                  Par exemple.
 372  <pre>
 373  class Pad {
 374      function Pad() { }
 375      function &amp;note($index) { }
 376  }
 377  </pre>
 378                  Dans ce cas, vous pouvez définir une référence dans la liste
 379                  des valeurs retournées par l'objet fantaisie...
 380  <pre>
 381  function testTaskReads() {
 382      $note = 'Buy books';
 383      $pad = new MockPad();
 384      $vector-&gt;<strong>returnsByReference(</strong>'note', $note, array(3)<strong>)</strong>;
 385      $task = new Task($pad);
 386      ...
 387  }
 388  </pre>
 389                  Avec cet assemblage vous savez qu'à chaque fois
 390                  que <span class="new_code">$pad-&gt;note(3)</span> est appelé
 391                  il renverra toujours la même <span class="new_code">$note</span>,
 392                  même si celle-ci est modifiée.
 393              </p>
 394              <p>
 395                  Ces trois facteurs, timing, paramètres et références,
 396                  peuvent être combinés orthogonalement.
 397                  Par exemple...
 398  <pre>
 399  $buy_books = 'Buy books';
 400  $write_code = 'Write code';
 401  $pad = new MockPad();
 402  $vector-&gt;<strong>returnsByReferenceAt(0, 'note', $buy_books, array('*', 3));</strong>
 403  $vector-&gt;<strong>returnsByReferenceAt(1, 'note', $write_code, array('*', 3));</strong>
 404  </pre>
 405                  Cela renverra une référence à <span class="new_code">$buy_books</span> et
 406                  ensuite à <span class="new_code">$write_code</span>, mais seuleent si deux paramètres
 407                  sont utilisés, le deuxième devant être l'entier 3.
 408                  Cela devrait couvrir la plupart des situations.
 409              </p>
 410              <p>
 411                  Un dernier cas délicat reste : celui d'un objet créant
 412                  un autre objet, plus connu sous le patron de conception "fabrique"
 413                  (ou "factory").
 414                  Supponsons qu'à la réussite d'une requête à
 415                  notre base de données imaginaire, un jeu d'enregistrements
 416                  est renvoyé sous la forme d'un itérateur, où chaque appel
 417                  au <span class="new_code">next()</span> sur notre itérateur donne une ligne
 418                  avant de s'arrêtre avec un false.
 419                  Cela semble à un cauchemar à simuler, alors qu'en fait un objet
 420                  fantaisie peut être créé avec les mécansimes ci-dessus...
 421  <pre>
 422  Mock::generate('DatabaseConnection');
 423  Mock::generate('ResultIterator');
 424  
 425  class DatabaseTest extends UnitTestCase {
 426  
 427      function testUserFinderReadsResultsFromDatabase() {<strong>
 428          $result = new MockResultIterator();
 429          $result-&gt;returns('next', false);
 430          $result-&gt;returnsAt(0, 'next', array(1, 'tom'));
 431          $result-&gt;returnsAt(1, 'next', array(3, 'dick'));
 432          $result-&gt;returnsAt(2, 'next', array(6, 'harry'));
 433  
 434          $connection = new MockDatabaseConnection();
 435          $connection-&gt;returns('selectQuery', $result);</strong>
 436  
 437          $finder = new UserFinder(<strong>$connection</strong>);
 438          $this-&gt;assertIdentical(
 439                  $finder-&gt;findNames(),
 440                  array('tom', 'dick', 'harry'));
 441      }
 442  }
 443  </pre>
 444                  Désormais ce n'est que si notre <span class="new_code">$connection</span>
 445                  est appelée par la méthode <span class="new_code">query()</span>
 446                  que sera retourné le <span class="new_code">$result</span>,
 447                  lui-même s'arrêtant au troisième appel
 448                  à <span class="new_code">next()</span>.
 449                  Ce devrait être suffisant comme information
 450                  pour que notre classe <span class="new_code">UserFinder</span>,
 451                  la classe effectivement testée ici,
 452                  fasse son boulot.
 453                  Un test très précis et toujours pas
 454                  de base de données en vue.
 455              </p>
 456              <p>
 457                  Nous pourrsion affiner ce test encore plus
 458                  en insistant pour que la bonne requête
 459                  soit envoyée...
 460  <pre>
 461  $connection-&gt;returns('selectQuery', $result, array(<strong>'select name, id from people'</strong>));
 462  </pre>
 463                  Là, c'est une mauvaise idée.
 464              </p>
 465              <p>
 466                  Niveau objet, nous avons un <span class="new_code">UserFinder</span>
 467                  qui parle à une base de données à travers une interface géante -
 468                  l'ensemble du SQL.
 469                  Imaginez si nous avions écrit un grand nombre de tests
 470                  qui dépendrait désormais de cette requête SQL précise.
 471                  Ces requêtes pourraient changer en masse pour tout un tas
 472                  de raisons non liés à ce test spécifique.
 473                  Par exemple, la règle pour les quotes pourrait changer,
 474                  un nom de table pourrait évoluer, une table de liaison pourrait
 475                  être ajouté, etc.
 476                  Cela entrainerait une ré-écriture de tous les tests à chaque fois
 477                  qu'un remaniement est fait, alors même que le comportement lui
 478                  n'a pas bougé.
 479                  Les tests sont censés aider au remaniement, pas le bloquer.
 480                  Pour ma part, je préfère avoir une suite de tests qui passent
 481                  quand je fais évoluer le nom des tables.
 482              </p>
 483              <p>
 484                  Et si vous voulez une règle, c'est toujours mieux de ne pas
 485                  créer un objet fantaisie sur une grosse interface.
 486              </p>
 487              <p>
 488                  Par contrast, voici le test complet...
 489  <pre>
 490  class DatabaseTest extends UnitTestCase {<strong>
 491      function setUp() { ... }
 492      function tearDown() { ... }</strong>
 493  
 494      function testUserFinderReadsResultsFromDatabase() {
 495          $finder = new UserFinder(<strong>new DatabaseConnection()</strong>);
 496          $finder-&gt;add('tom');
 497          $finder-&gt;add('dick');
 498          $finder-&gt;add('harry');
 499          $this-&gt;assertIdentical(
 500                  $finder-&gt;findNames(),
 501                  array('tom', 'dick', 'harry'));
 502      }
 503  }
 504  </pre>
 505                  Ce test est immunisé contre le changement de schéma.
 506                  Il échouera uniquement si vous changez la fonctionnalité,
 507                  ce qui correspond bien à ce qu'un test doit faire.
 508              </p>
 509              <p>
 510                  Il faut juste faire attention à ces méthodes <span class="new_code">setUp()</span>
 511                  et <span class="new_code">tearDown()</span> que nous avons survolé pour l'instant.
 512                  Elles doivent vider les tables de la base de données
 513                  et s'assurer que le schéma est bien défini.
 514                  Cela peut se engendrer un peu de travail supplémentaire,
 515                  mais d'ordinaire ce type de code existe déjà - à commencer pour
 516                  le déploiement.
 517              </p>
 518              <p>
 519                  Il y a au moins un endroit où vous aurez besoin d'objets fantaisie :
 520                  c'est la simulation des erreurs.
 521                  Exemple, la base de données tombe pendant que <span class="new_code">UserFinder</span>
 522                  fait son travail. Le <span class="new_code">UserFinder</span> se comporte-t-il bien ?
 523  <pre>
 524  class DatabaseTest extends UnitTestCase {
 525  
 526      function testUserFinder() {
 527          $connection = new MockDatabaseConnection();<strong>
 528          $connection-&gt;throwOn('selectQuery', new TimedOut('Ouch!'));</strong>
 529          $alert = new MockAlerts();<strong>
 530          $alert-&gt;expectOnce('notify', 'Database is busy - please retry');</strong>
 531          $finder = new UserFinder($connection, $alert);
 532          $this-&gt;assertIdentical($finder-&gt;findNames(), array());
 533      }
 534  }
 535  </pre>
 536                  Nous avons transmis au <span class="new_code">UserFinder</span>
 537                  un objet <span class="new_code">$alert</span>.
 538                  Il va transmettre un certain nombre de belles notifications
 539                  à l'interface utilisatuer en cas d'erreur.
 540                  Nous préfèrerions éviter de saupoudrer notre code avec
 541                  des commandes <span class="new_code">print</span> codées en dur si nous pouvons
 542                  l'éviter.
 543                  Emballer les moyens de sortie veut dire que nous pouvons utiliser
 544                  ce code partout. Et cela rend notre code plus facile.
 545              </p>
 546              <p>
 547                  Pour faire passer ce test, le finder doit écrire un message sympathique
 548                  et compréhensible à l'<span class="new_code">$alert</span>, plutôt que de propager
 549                  l'exception. Jusque là, tout va bien.
 550              </p>
 551              <p>
 552                  Comment faire en sorte que la <span class="new_code">DatabaseConnection</span> fantaisie
 553                  soulève une exception ?
 554                  Nous la générons avec la méthode <span class="new_code">throwOn</span> sur l'objet fantaisie.
 555              </p>
 556              <p>
 557                  Enfin, que se passe-t-il si la méthode voulue pour la simulation
 558                  n'existe pas encore ?
 559                  Si vous définissez une valeur de retour sur une méthode absente,
 560                  alors SimpleTest vous répondra avec une erreur.
 561                  Et si vous utilisez <span class="new_code">__call()</span> pour simuler
 562                  des méthodes dynamiques ?
 563              </p>
 564              <p>
 565                  Les objets avec des interfaces dynamiques, avec <span class="new_code">__call</span>,
 566                  peuvent être problématiques dans l'implémentation courante
 567                  des objets fantaisie.
 568                  Vous pouvez en créer un autour de la méthode <span class="new_code">__call()</span>
 569                  mais c'est très laid.
 570                  Et pourquoi un test devrait connaître quelque chose avec un niveau
 571                  si bas dans l'implémentation. Il n'a besoin que de simuler l'appel.
 572              </p>
 573              <p>
 574                  Il y a bien moyen de contournement : ajouter des méthodes complémentaires
 575                  à l'objet fantaisie à la génération.
 576  <pre>
 577  <strong>Mock::generate('DatabaseConnection', 'MockDatabaseConnection', array('setOptions'));</strong>
 578  </pre>
 579                  Dans une longue suite de tests cela pourrait entraîner des problèmes,
 580                  puisque vous avez probablement déjà une version fantaisie
 581                  de la classe appellée <span class="new_code">MockDatabaseConnection</span>
 582                  sans les méthodes complémentaires.
 583                  Le générateur de code ne générera pas la version fantaisie de la classe
 584                  s'il en existe déjà une version avec le même nom.
 585                  Il vous deviendra impossible de déterminer où est passée votre méthode
 586                  si une autre définition a été lancé au préalable.
 587              </p>
 588              <p>
 589                  Pour pallier à ce problème, SimpleTest vous permet de choisir
 590                  n'importe autre nom pour la nouvelle classe au moment même où
 591                  vous ajouter les méthodes complémentaires.
 592  <pre>
 593  Mock::generate('DatabaseConnection', <strong>'MockDatabaseConnectionWithOptions'</strong>, array('setOptions'));
 594  </pre>
 595                  Ici l'objet fantaisie se comportera comme si
 596                  <span class="new_code">setOptions()</span> existait bel et bien
 597                  dans la classe originale.
 598              </p>
 599              <p>
 600                  Les objets fantaisie ne peuvent être utilisés qu'à l'intérieur
 601                  des scénarios de test, étant donné qu'à l'apparition d'une attente
 602                  ils envoient des messages directement au scénario de test courant.
 603                  Les créer en dehors d'un scénario de test entraînera une erreur
 604                  de run time quand une attente est déclenchée et qu'il n'y a pas
 605                  de scénario de test en cours pour recevoir le message.
 606                  Nous pouvons désormais couvrir ces attentes.
 607              </p>
 608          
 609          <h2>
 610  <a class="target" name="expectations"></a>Objets fantaisie en tant que critiques</h2>
 611              <p>
 612                  Même si les bouchons serveur isolent vos tests des perturbations
 613                  du monde réel, ils n'apportent que le moitié des bénéfices possibles.
 614                  Vous pouvez obtenir une classe de test qui reçoive les bons messages,
 615                  mais cette nouvelle classe envoie-t-elle les bons ?
 616                  Le tester peut devenir très bordélique sans
 617                  une librairie d'objets fantaise.
 618              </p>
 619              <p>
 620                  Voici un exemple, prenons une classe <span class="new_code">PageController</span>
 621                  toute simple qui traitera un formulaire de paiement
 622                  par carte bleue...
 623  <pre>
 624  class PaymentForm extends PageController {
 625      function __construct($alert, $payment_gateway) { ... }
 626      function makePayment($request) { ... }
 627  }
 628  </pre>
 629                  Cette classe a besoin d'un <span class="new_code">PaymentGateway</span>
 630                  pour parler concrètement à la banque.
 631                  Il utilise aussi un objet <span class="new_code">Alert</span>
 632                  pour gérer les erreurs.
 633                  Cette dernière classe parle à la page ou au template.
 634                  Elle est responsable de l'affichage du message d'erreur
 635                  et de la mise en valeur de tout champ du formulaire
 636                  qui serait incorrecte.
 637              </p>
 638              <p>
 639                  Elle pourrait ressembler à...
 640  <pre>
 641  class Alert {
 642      function warn($warning, $id) {
 643          print '&lt;div class="warning"&gt;' . $warning . '&lt;/div&gt;';
 644          print '&lt;style type="text/css"&gt;#' . $id . ' { background-color: red }&lt;/style&gt;';
 645      }
 646  }
 647  </pre>
 648              </p>
 649              <p>
 650                  Portons notre attention à la méthode <span class="new_code">makePayment()</span>.
 651                  Si nous n'inscrivons pas un numéro "CVV2" (celui à trois
 652                  chiffre au dos de la carte bleue), nous souhaitons afficher
 653                  une erreur plutôt que d'essayer de traiter le paiement. 
 654                  En mode test...
 655  <pre>
 656  &lt;?php
 657  require_once('simpletest/autorun.php');
 658  require_once('payment_form.php');
 659  Mock::generate('Alert');
 660  Mock::generate('PaymentGateway');
 661  
 662  class PaymentFormFailuresShouldBeGraceful extends UnitTestCase {
 663  
 664      function testMissingCvv2CausesAlert() {
 665          $alert = new MockAlert();
 666          <strong>$alert-&gt;expectOnce(
 667                      'warn',
 668                      array('Missing three digit security code', 'cvv2'));</strong>
 669          $controller = new PaymentForm(<strong>$alert</strong>, new MockPaymentGateway());
 670          $controller-&gt;makePayment($this-&gt;requestWithMissingCvv2());
 671      }
 672  
 673      function requestWithMissingCvv2() { ... }
 674  }
 675  ?&gt;
 676  </pre>
 677                  Première question : où sont passés les assertions ?
 678              </p>
 679              <p>
 680                  L'appel à <span class="new_code">expectOnce('warn', array(...))</span> annonce
 681                  à l'objet fantaisie qu'il faut s'attendre à un appel à <span class="new_code">warn()</span>
 682                  avant la fin du test.
 683                  Quand il débouche sur l'appel à <span class="new_code">warn()</span>, il vérifie
 684                  les arguments. Si ceux-ci ne correspondent pas, alors un échec
 685                  est généré. Il échouera aussi si la méthode n'est jamais appelée.
 686              </p>
 687              <p>
 688                  Non seulement le test ci-dessus s'assure que <span class="new_code">warn</span>
 689                  a bien été appelé, mais en plus qu'il a bien reçu la chaîne
 690                  de caractère "Missing three digit security code"
 691                  et même le tag "cvv2".
 692                  L'équivalent de <span class="new_code">assertIdentical()</span> est appliqué
 693                  aux deux champs quand les paramètres sont comparés.
 694              </p>
 695              <p>
 696                  Si le contenu du message vous importe peu, surtout dans le cas
 697                  d'une interface utilisateur qui change régulièrement,
 698                  nous pouvons passer ce paramètre avec l'opérateur "*"...
 699  <pre>
 700  class PaymentFormFailuresShouldBeGraceful extends UnitTestCase {
 701  
 702      function testMissingCvv2CausesAlert() {
 703          $alert = new MockAlert();
 704          $alert-&gt;expectOnce('warn', array(<strong>'*'</strong>, 'cvv2'));
 705          $controller = new PaymentForm($alert, new MockPaymentGateway());
 706          $controller-&gt;makePayment($this-&gt;requestWithMissingCvv2());
 707      }
 708  
 709      function requestWithMissingCvv2() { ... }
 710  }
 711  </pre>
 712                  Nous pouvons même rendre le test encore moins spécifique
 713                  en supprimant complètement la liste des paramètres...
 714  <pre>
 715  function testMissingCvv2CausesAlert() {
 716      $alert = new MockAlert();
 717      <strong>$alert-&gt;expectOnce('warn');</strong>
 718      $controller = new PaymentForm($alert, new MockPaymentGateway());
 719      $controller-&gt;makePayment($this-&gt;requestWithMissingCvv2());
 720  }
 721  </pre>
 722                  Ceci vérifiera uniquement si la méthode a été appelé,
 723                  ce qui est peut-être un peu drastique dans ce cas.
 724                  Plus tard, nous verrons comment alléger les attentes
 725                  plus précisement.
 726              </p>
 727              <p>
 728                  Des tests sans assertions peuvent être à la fois compacts
 729                  et très expressifs. Parce que nous interceptons l'appel
 730                  sur le chemin de l'objet, ici de classe <span class="new_code">Alert</span>,
 731                  nous évitons de tester l'état par la suite.
 732                  Cela évite les assertions dans les tests, mais aussi
 733                  l'obligation d'ajouter des accesseurs uniquement
 734                  pour les tests dans le code original.
 735                  Si vous en arrivez à ajouter des accesseurs de ce type,
 736                  on parle alors de "state based testing" dans le jargon
 737                  ("test piloté par l'état"),
 738                  il est probablement plus que temps d'utiliser
 739                  des objets fantaisie dans vos tests.
 740                  On peut alors parler de "behaviour based testing"
 741                  (ou "test piloté par le comportement") :
 742                  c'est largement mieux !
 743              </p>
 744              <p>
 745                  Ajoutons un autre test.
 746                  Assurons nous que nous essayons même pas un paiement sans CVV2...
 747  <pre>
 748  class PaymentFormFailuresShouldBeGraceful extends UnitTestCase {
 749  
 750      function testMissingCvv2CausesAlert() { ... }
 751  
 752      function testNoPaymentAttemptedWithMissingCvv2() {
 753          $payment_gateway = new MockPaymentGateway();
 754          <strong>$payment_gateway-&gt;expectNever('pay');</strong>
 755          $controller = new PaymentForm(new MockAlert(), $payment_gateway);
 756          $controller-&gt;makePayment($this-&gt;requestWithMissingCvv2());
 757      }
 758  
 759      ...
 760  }
 761  </pre>
 762                  Vérifier une négation peut être très difficile
 763                  dans les tests, mais <span class="new_code">expectNever()</span>
 764                  rend l'opération très facile heureusement.
 765              </p>
 766              <p>
 767                  <span class="new_code">expectOnce()</span> et <span class="new_code">expectNever()</span> sont
 768                  suffisants pour la plupart des tests, mais
 769                  occasionnellement vous voulez tester plusieurs évènements.
 770                  D'ordinaire pour des raisons d'usabilité, nous souhaitons
 771                  que tous les champs manquants du formulaire soient
 772                  mis en relief, et pas uniquement le premier.
 773                  Cela veut dire que nous devrions voir de multiples appels
 774                  à <span class="new_code">Alert::warn()</span>, pas juste un...
 775  <pre>
 776  function testAllRequiredFieldsHighlightedOnEmptyRequest() {
 777      $alert = new MockAlert();<strong>
 778      $alert-&gt;expectAt(0, 'warn', array('*', 'cc_number'));
 779      $alert-&gt;expectAt(1, 'warn', array('*', 'expiry'));
 780      $alert-&gt;expectAt(2, 'warn', array('*', 'cvv2'));
 781      $alert-&gt;expectAt(3, 'warn', array('*', 'card_holder'));
 782      $alert-&gt;expectAt(4, 'warn', array('*', 'address'));
 783      $alert-&gt;expectAt(5, 'warn', array('*', 'postcode'));
 784      $alert-&gt;expectAt(6, 'warn', array('*', 'country'));
 785      $alert-&gt;expectCallCount('warn', 7);</strong>
 786      $controller = new PaymentForm($alert, new MockPaymentGateway());
 787      $controller-&gt;makePayment($this-&gt;requestWithMissingCvv2());
 788  }
 789  </pre>
 790                  Le compteur dans <span class="new_code">expectAt()</span> précise
 791                  le nombre de fois que la méthode a déjà été appelée.
 792                  Ici nous vérifions que chaque champ sera bien mis en relief.
 793              </p>
 794              <p>
 795                  Notez que nous sommes forcé de tester l'ordre en même temps.
 796                  SimpleTest n'a pas encore de moyen pour éviter cela,
 797                  mais dans une version future ce sera corrigé.
 798              </p>
 799              <p>
 800                  Voici la liste complètes des attentes
 801                  que vous pouvez préciser sur une objet fantaisie
 802                  dans <a href="http://simpletest.org/">SimpleTest</a>.
 803                  Comme pour les assertions, ces méthodes prennent en option
 804                  un message d'erreur.
 805                  <table>
 806                      <thead><tr>
 807  <th>Attente</th>
 808  <th>Description</th>
 809  </tr></thead>
 810                      <tbody>
 811                          <tr>
 812                              <td><span class="new_code">expect($method, $args)</span></td>
 813                              <td>Les arguements doivent correspondre si appelés</td>
 814                          </tr>
 815                          <tr>
 816                              <td><span class="new_code">expectAt($timing, $method, $args)</span></td>
 817                              <td>Les arguements doiven correspondre si appelés lors du passage numéro <span class="new_code">$timing</span>
 818  </td>
 819                          </tr>
 820                          <tr>
 821                              <td><span class="new_code">expectCallCount($method, $count)</span></td>
 822                              <td>La méthode doit être appelée exactement <span class="new_code">$count</span> fois</td>
 823                          </tr>
 824                          <tr>
 825                              <td><span class="new_code">expectMaximumCallCount($method, $count)</span></td>
 826                              <td>La méthode ne doit pas être appelée plus de <span class="new_code">$count</span> fois</td>
 827                          </tr>
 828                          <tr>
 829                              <td><span class="new_code">expectMinimumCallCount($method, $count)</span></td>
 830                              <td>La méthode ne doit pas être appelée moins de <span class="new_code">$count</span> fois</td>
 831                          </tr>
 832                          <tr>
 833                              <td><span class="new_code">expectNever($method)</span></td>
 834                              <td>La méthode ne doit jamais être appelée</td>
 835                          </tr>
 836                          <tr>
 837                              <td><span class="new_code">expectOnce($method, $args)</span></td>
 838                              <td>La méthode ne doit être appelée qu'une seule fois et avec les arguments (en option)</td>
 839                          </tr>
 840                          <tr>
 841                              <td><span class="new_code">expectAtLeastOnce($method, $args)</span></td>
 842                              <td>La méthode doit être appelée au moins une seule fois et toujours avec au moins un des arguments attendus</td>
 843                          </tr>
 844                      </tbody>
 845                  </table>
 846                  Où les paramètres sont...
 847                  <dl>
 848                      <dt class="new_code">$method</dt>
 849                      <dd>
 850                          Le nom de la méthode, sous la forme d'une chaîne de caractères,
 851                          à laquelle il faut appliquer la condition.
 852                      </dd>
 853                      <dt class="new_code">$args</dt>
 854                      <dd>
 855                          Les argumetns sous la forme d'une liste.
 856                          Les jokers peuvent être inclus de la même manière
 857                          que pour <span class="new_code">setReturn()</span>.
 858                          Cet argument est optionnel pour <span class="new_code">expectOnce()</span>
 859                          et <span class="new_code">expectAtLeastOnce()</span>.
 860                      </dd>
 861                      <dt class="new_code">$timing</dt>
 862                      <dd>
 863                          La seule marque dans le temps pour tester la condition.
 864                          Le premier appel commence à zéro et le comptage se fait
 865                          séparement sur chaque méthode.
 866                      </dd>
 867                      <dt class="new_code">$count</dt>
 868                      <dd>Le nombre d'appels attendu.</dd>
 869                  </dl>
 870              </p>
 871              <p>
 872                  Si vous n'avez qu'un seul appel dans votre test, assurez vous
 873                  d'utiliser <span class="new_code">expectOnce</span>.<br>
 874                  Utiliser <span class="new_code">$mocked-&gt;expectAt(0, 'method', 'args);</span>
 875                  tout seul ne permettra qu'à la méthode de ne jamais être appelée.
 876                  Vérifier les arguements et le comptage total sont pour le moment
 877                  indépendants.
 878                  Ajouter une attente <span class="new_code">expectCallCount()</span> quand
 879                  vous utilisez <span class="new_code">expectAt()</span> (dans le cas sans appel)
 880                  est permis.
 881              </p>
 882              <p>
 883                  Comme les assertions à l'intérieur des scénarios de test,
 884                  toutes ces attentes peuvent incorporer une surchage
 885                  sur le message sous la forme d'un paramètre supplémentaire.
 886                  Par ailleurs le message original peut être inclus dans la sortie
 887                  avec "%s".
 888              </p>
