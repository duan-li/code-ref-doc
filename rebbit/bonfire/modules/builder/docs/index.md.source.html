   1  ## Module Builder
   2  
   3  Module Builder is available in the Developer context and requires some knowledge of database table design, column types and form layout. It is intended to create some skeleton code for you that you can then continue working with. It cannot modify existing files and will overwrite files if you try.
   4  
   5  ## Building a module
   6  
   7  The builder generates the code for a module based on the inputs to a few form fields and can build a DB based module or a module without DB access.  The builder can create a DB table for you or you can use an existing db table to create the module.
   8  
   9  To create a new module which does not require DB access requires you to fill out one form field (the module name field) and pressing the "Build" button.
  10  
  11  This will generate a module in the *application/modules* folder with config, controllers, language, migrations (for permissions) and view files.  Amazing for one form field and a button click.
  12  
  13  Of course that's just the basic module.  To get more you have to give more!
  14  
  15  If you want to use a DB table then you need to fill out more fields.  You can choose to build a new DB table or choose an existing table.
  16  
  17  <a name="new-db-table"></a>
  18  ### With a new database table
  19  
  20  You must choose the number of fields you would like in the table, excluding the primary key field which is automatically included.
  21  
  22  Then for every field you fill in the form Label, table field name, form input element, table field type, table field length and the form validation settings you would like.
  23  
  24  The primary key is assumed to be "id" but this is editable before (and of course after) the form is submitted.
  25  
  26  Once the field details have been entered then just click the "Build" button.
  27  
  28  Hey presto! Now you have a brand new shiny module with all files mentioned above as well as a new model file and a migration file to create the table in the database.  The migration file is even run automatically so you can use the module straight away.
  29  
  30  ### With an existing database table
  31  
  32  You must choose the name of the table which you would like to use to build your module.
  33  
  34  The table name must include the prefix used for your Bonfire site which is "bf_" by default. If your table does not have this prefix then please rename it to include the prefix. Now click on the "Build" button.
  35  
  36  This  reads the structure of the database table and picks out the field characteristics including the primary field.
  37  
  38  The page returned includes settings for each of the fields and sets up the field label for you.
  39  
  40  For every field you can change this data but it is advisable to only change the Label and Validation settings as anything else may cause errors.
  41  
  42  
  43  When you are finished editing the fields just click the "Build" button again.
  44  
  45  Now you have a new module based on your previous database table with all files mentioned above as well as a new model file.  The migration file is not created in this case as the table already exists but there is a migration file for the permission settings which is processed automatically so you can use the module straight away.
  46

  48  ## Module Builder Options
  49  
  50  There are lots of options which you can use when building your module.  These options are hidden by default but to see them you just click on the "Toggle Advanced Options" links.
  51  
  52  Clicking on the link will display the options below the link. Then for every field you fill in the form Label, table field name, form input element, table field type, table field length and the form validation settings you would like.
  53  
  54  There are two sets of options, Module Options and Table Options which we will describe below.
  55  
  56  <a name="module-options"></a>
  57  ### Module Options
  58  
  59  The Module Options allow you to customize the module functionality.  The options are:
  60  
  61  * **Module Description** - The module description is placed in the module config file and appears on the module list in the main Module Builder page.
  62  * **Contexts Required** - Lists the available Bonfire contexts, you choose the contexts you would like the builder to create for your module. All contexts are selected by default.
  63  * **Controller Actions** - Lists the CRUD actions which the builder can create for you. All actions are selected by default.
  64  * **Give Role Full Access** - Choose which Bonfire role you would like to have full access to the module after creation.  The Administrator role is the default.
  65  
  66  
  67  <a name="table-options"></a>
  68  ### Table Options
  69  
  70  The Table Options are related to the DB table if you chose to build the module with DB access. The options are:
  71  
  72  * **Table Name** - This is the name given to the table in the database. This is defaulted to the lowercase and underscored version of the module name.
  73  * **Form Error Delimiters**
  74  * **Use "Soft" Deletes** - Default value: FALSE
  75  * **Use "Created" Field** - Default value: FALSE
  76  * **"Created_on" Field Name** - Default value: 'created_on'
  77  * **Use "Modified" Field** - Default value: FALSE
  78  * **"Modified" Field Name** - Default value: 'modified_on'
  79  * **Primary Key** - Default value: 'id'
  80  
  81  <a name="known-issues"></a>
  82  ## Known Issues
  83  
  84  The Module Builder is a great way to get started but there are a couple of issues which you should be aware of:
  85  
  86  * Error messaging is not the best when something goes wrong.
  87  * Building from existing tables shows some fields in the form which are not applicable.
