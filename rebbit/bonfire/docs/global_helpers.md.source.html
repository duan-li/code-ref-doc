   1  Bonfire provides a number of resources that are loaded globally and are intended to make life a little easier for you. Most of these are loaded from the *application/helpers/application_helper.php* file.
   2  
   3  ### <tt>$current_user</tt>
   4  
   5  If a user is logged in, you can access the current user's detail within any of your controllers that extend Bonfire's provided controllers, like Front_Controller, Admin_Controller, etc. This is accessed by <tt>$this->current_user</tt>.
   6  
   7  This is an object with the user's details:
   8  
   9  
  10      stdclass Object
  11      {
  12          [id]        => 1
  13          [role_id]   => 1
  14          [email]     => darth@empire.com
  15          [username]  => Darth Vader
  16          [password_hash] => ...
  17          [reset_hash]    => ...
  18          [last_login]    =>
  19          [last_ip]       =>
  20          [created_on]    =>
  21          [deleted]       => 0
  22          [banned]        => 0
  23          [ban_message]   =>
  24          [reset_by]      =>
  25          [display_name]  => Darth
  26          [display_name_changed]  =>
  27          [timezone]      => UM6
  28          [language]      =>
  29          [active]        => 1
  30          [activate_hash] =>
  31          [password_iterations]   => 8
  32          [force_password_reset]  => 0
  33          [role_name]     => Administrator
  34          [user_img]      => (url for their avatar)
  35      }
  36  
  37  
  38  The exact details may change over the versions, but the entire <tt>users</tt> table record is provided to you.
  39  
  40  This same data is made available within all of your views as <tt>$current_user</tt>.
  41  
  42  
  43      <?php echo $current_user->display_name ?>

  48  ### <tt>e()</tt>
  49  
  50  A convenience function that can (and probably should) be used to replace <tt>echo()</tt> in any places that you output text that a user might have entered. This function helps to defeat XSS attacks by running the text through <tt>htmlspecialchars()</tt>.
  51  
  52  
  53      <?php e($user->display_name); ?>
  54

  58  ### <tt>js_escape()</tt>
  59  
  60  Performs some simple escaping of strings that will be inserted into javascript functions, like <tt>confirm()</tt> or <tt>alert()</tt>. Not suitable for use by itself within document.write. The only parameter is the string to escape.

  72  ### <tt>gravatar_link()</tt>
  73  
  74  Returns a URL for the user from [Gravatar](http://gravatar.com) and provides a default image (the [identicon](http://en.gravatar.com/site/implement/images/)) when the user hasn't provided one. This is intended to be used within your views for displaying the user's avatar.
  75  
  76  
  77      <?php echo gravatar_image($current_user->email, 48, $current_user->display_name, $current_user->display_name, $class, $id); ?>
  78  
  79      // Returns:
  80      <img src="http://gravatar.com/avatar/XXX?s=48&r=pg&d=identicon" width="48" height="48" alt="" title="" />
  81  
  82  
  83  The first parameter is the email address for the user. This is how they are referenced at Gravatar. The second parameter is the size of the image in pixels. The third parameter is the alt tag and the fourth parameter is the title for the tag. The fifth and sixth parameters are the class and id to be given to the img, respectively.

  87  ### <tt>logit()</tt>
  88  
  89  Logs an error to the Console in the profiler (if loaded) and to the log files.
  90  
  91  
  92      logit('Some debugging message', 'debug');
  93  
  94  
  95  The first parameter is the message to log. The second parameter is the log level.
  96  
  97  If the profiler is enabled and the Console class is loaded, which it is by default in development mode, the string will show up in the Console section of the profiler.
  98  
  99  If logging is enabled in your <tt>application/config/config.php</tt> file, then it will also be logged to your default log file.

 103  ### <tt>module_folders()</tt>

 117  ### <tt>module_list</tt>
 118  
 119  Returns a list of all modules in the system. If TRUE is passed as the only parameter, any Bonfire core modules will NOT be shown, only your custom modules. If FALSE is passed, both core and custom modules will be listed.
 120  
 121

 145  ### <tt>module_controller_exists()</tt>
 146  
 147  Looks within a module to see if a certain controller exists. The first parameter is the name of the controller, and the second parameter is the module name.
 148  
 149  It returns either TRUE or FALSE.
 150  
 151  
 152      if (module_controller_exists('content', 'users')) { . . . }

 157  ### <tt>module_file_path()</tt>
 158  
 159  Locates a file within a module and returns the path to that file. The first parameter is the name of the module. The second parameter is the name of the folder. The last parameter is the name of the file that you're looking for (including the extension).
 160  
 161  It returns the full server path to the file, if found.
 162  
 163  
 164      $path = module_file_path('users', 'assets', 'js/users.js');

 169  ### <tt>module_path()</tt>
 170  
 171  Returns the full server path to a module and, optionally, a folder within that module. The first parameter is the name of the module. The second parameter is the name of the folder.
 172  
 173  
 174      $path = module_path('users', 'assets');

 179  ### <tt>module_files()</tt>
 180  
 181  Returns an associative array of files within one or more modules.
 182

 216                          (
 217                              [0] => sysinfo_lang.php
 218                          )

 250  ### <tt>module_config()</tt>
 251  
 252  Returns the 'module_config' array from a module's <tt>config/config.php</tt> file. The <tt>module_config</tt> contains more information about a module, like the author, menu behavior, etc.
