   1  ## What Are Assets?
   2  
   3  Bonfire considers any CSS files, javascript files, and many image files to be Assets. While we understand that projects will often contain Flash files or other types of files that would be considered 'assets', Bonfire only accounts for these three types of files but does provide some helper methods to assist with other types of files.
   4  
   5  Assets files can be stored in an application-wide repository, in your theme, or with each module. Because of the possibility of some of these files not being web-accessible, module-related assets are always combined and cached in a web-accessible folder.
   6  
   7  Assets can be located in one of several locations:
   8  
   9      /bonfire/modules/my_module/assets
  10      /bonfire/themes/active_theme/assets
  11      /bonfire/themes/default_theme/assets
  12      /assets
  13  
  14  
  15  Whenever you use one of the asset rendering methods, like <tt>css()</tt>, <tt>js()</tt>, and <tt>image()</tt>, it will search in each of those folders (as well as a folder named by the asset type under those folders, ie. 'css') in the order listed above. Note that it only looks within the module if it was added via <tt>add_module_css()</tt> or <tt>add_module_js()</tt>.
  16  
  17  <a name="asset-base"></a>
  18  ### Customizing Asset Base Folder
  19  
  20  If you need to, you can customize the folders that your assets are located in via settings in the <tt>bonfire/application/config/application.php</tt> configuration file.
  21  
  22  To change the primary folder that all of your assets are located in, you can set the <tt>assets.base_folder</tt> setting. Note that this is not recommended as it may cause instability in the application. _Testing has not been done to ensure that this is a viable option at the moment._
  23  
  24  
  25      $config['assets.base_folder'] = 'assets';
  26  
  27  
  28  <a name="asset-folders"></a>
  29  ### Customizing Asset Folders
  30  
  31  Within the base folder (above), each asset is organized by asset type. By default, the folders are named <tt>js</tt>, <tt>css</tt>, and <tt>images</tt>. This can be changed in the application config file. Again, though this has not been tested to ensure this works. Changing this value might render your site unusable.
  32  
  33  
  34      $config['assets.asset_folders'] = array(
  35          'css'   => 'css',
  36          'js'    => 'js',
  37          'image' => 'images'
  38      );
  39  
  40  <a name="css-main"></a>
  41  ## Working With CSS
  42  
  43  Stylesheets can be hosted with the application or on other servers. However, when it comes time to tweak your site's front-end performance, only those files that are hosted locally with the site can be combined and minified successfully. It is recommended that you always use the raw stylesheets (not minified) to take full advantage of the minification abilities, as minifying a CSS file twice might corrupt the styles.
  44  
  45  <a name="css"></a>
  46  ### <tt>css()</tt>
  47  
  48  This method provides a simple way to render all of your CSS links in one method call. It can be used in two different fashions: to output a single link tag, or to output link tags to all stylesheets that it knows about.
  49  
  50  Whenever a relative path is provided, it is assumed to be relative to either the main folder being searched in (ie. /assets) or the type-specific folder (ie. /assets/css).
  51  
  52  When called with no parameters, the <tt>css()</tt> method will create links for all stylesheets that have been added to it via either the <tt>add_css()</tt> or <tt>add_module_css()</tt> methods, below.
  53  
  54  
  55      Assets::css();
  56  
  57      // Creates:
  58      <link rel="stylesheet" type="text/css" href="http://mysite.com/themes/default/screen.css" media="screen" />
  59  
  60  
  61  **Media Type**
  62  
  63  You can specify the media type for your CSS to render, by passing it as the second parameter. The default value is 'screen'.
  64  
  65  
  66      Assets::css(null, 'screen');
  67  
  68  
  69  **Bypassing Inheritance**
  70  
  71  Bonfire's Assets system is designed with a Parent/Child theme framework in mind. That means that you can quickly build a new theme that inherits all of a default (or parent) theme and only changing the files that need to be changed. This applies to both assets and layout files.
  72  
  73  If you're using a theme that you've created called 'darthvader', it will look for files of the same name in both the _darthvader_ theme and the _default_ theme. If you do not want it to look for and include these files, you can pass in TRUE as the third parameter.
  74  
  75  
  76      Assets::css(null, 'screen', TRUE);
  77  
  78  
  79  **Global Files**
  80  
  81  A couple of different files will automatically appear within your styles, without you needing to do any work.
  82  
  83  The Assets library will search for a file within your theme for a file named the same as the media that you're currently rendering. By default, this is the <tt>bonfire/themes/my_theme/screen.css</tt> file.
  84  
  85  It will also look for any stylesheets within your theme that match the name of the controller that is active. So, if the **blog** controller is currently running, it will look for a file at <tt>bonfire/themes/my_theme/blog.css</tt>.
  86  
  87  **Rendering a Single File**
  88  
  89  When you just want to output a link to a single CSS file, you can use the <tt>css()</tt> method with a single parameter that is the name of the file to link to. The link can be either relative or absolute.
  90  
  91  
  92      echo Assets::css('http://mysite.com/path/to/my.css');
  93      // Output: <link rel="stylesheet" type="text/css" href="http://mysite.com/path/to/my.css" media="screen" />
  94  
  95      echo Assets::css('/path/to/my.css');
  96      // Output: <link rel="stylesheet" type="text/css" href="http://mysite.com/path/to/my.css" media="screen" />
  97  
  98  
  99  **Rendering Multiple Custom Files**
 100  
 101  If you want to render multiple links to CSS files, you can pass an array of stylesheets. These stylesheets will be merged with any styles that have been previously added via either <tt>add_css()</tt> or <tt>add_module_css()</tt>.
 102  
 103  
 104      $styles = array(
 105          'bootstrap.css',
 106          'markitup/sets/minimal/styles'
 107      );
 108  
 109      Assets::css($styles);
 110  
 111  
 112  <a name="add-css"></a>
 113  ### <tt>add_css()</tt>
 114  
 115  Adds additional file(s) to the links that are output with the <tt>css()</tt> method. The first parameters is the name of the file to be linked to. It should be relative to one of the paths described above. The second, optional, parameter is the media type. This defaults to 'screen'.
 116  
 117  
 118      Assets::add_css('blog.css');
 119  
 120  <a name="module-css"></a>
 121  ### 2.3 <tt>add_module_css()</tt>
 122  
 123  Adds a stylesheet that is part of a module to the styles to be output with the <tt>css()</tt> method. The first parameter is the name of the module. The second is the name of the file to be linked to. The third, optional, parameter is the media type this file belongs to. It defaults to 'screen'.
 124  
 125  
 126      Assets::add_module_css('blog', 'posts.css');
 127  
 128  
 129  Because module files are often inaccessible via a web browser, all module files are combined and cached in the <tt>assets/cache</tt> folder. The name of the file is built as follows for the maximum reuse:
 130  
 131      {theme name}_{module name}_{controller name}_mod.css
 132  
 133  The above blog/posts.css file would be in a file named <tt>default_blog_blog_mod.css</tt>.

 135  <a name="js-main"></a>
 136  ## Working With Javascript
 137  
 138  Script files can be hosted with the application or on other servers. However, when it comes time to tweak your site's front-end performance, only those files that are hosted locally with the site can be combined and minified successfully. It is recommended that you always use the raw scripts (not minified) to take full advantage of the minification abilities, as minifying a JS file twice might corrupt the script.
 139  
 140  For the sake of the Assets library, javascript files are broken up into 3 types of scripts:
 141  
 142  - **External** scripts are javascript files that included with a <tt>&lt;script src=""></script></tt> tag. These will typically be hosted with your site, but might be hosted on another server.
 143  - **Inline** scripts are scripts that are intended to be written out as part of the page's HTML. You may include a file as inline for performance reasons or simply because it only affects a single page on your site and including it in a larger external file does not make sense.
 144  - **Module** scripts are scripts that are part of a module within Bonfire. Because of security reasons, you often will not be to access the module files from the browser so these scripts are always combined and cached in the main <tt>assets/</tt> folder.
 145  
 146  
 147  <a name="js-js"></a>
 148  ### <tt>js()</tt>
 149  
 150  This method provides a single call to render all external, module, or inline script tags.
 151  
 152  Whenever a relative path is provided, it is assumed to be relative to either the main folder being searched in (ie. /assets) or the type-specific folder (ie. /assets/js).
 153  
 154  When called with no parameters, the <tt>js()</tt> method will create links/inline-code for all scripts that it knows about.
 155  
 156  
 157      echo Assets::js();
 158  
 159  
 160  Might produce:
 161  
 162  {% geshi "Javascript" use_classes %}
 163  <script src="http://bonfire.dev/assets/js/bootstrap.min.js" type="text/javascript" ></script>
 164  <script type="text/javascript">
 165  $(document).ready(function(){
 166      $(".dropdown-toggle").dropdown();$(".tooltips").tooltip();$(".login-btn").click(function(e){ e.preventDefault(); $("#modal-login").modal(); });
 167  });
 168  </script>
 169  ```
 170  
 171  Notice that the inline scripts is automatically wrapped with customizable open and close strings. In this case, it produced an appropriate wrapper for use with jQuery. This can be changed in the <tt>application/config/application.php</tt> file.
 172  
 173  
 174      $config['assets.js_opener'] = '$(document).ready(function(){'. "\n";
 175      $config['assets.js_closer'] = '});'. "\n";
 176  
 177  
 178  **Single Script Tags**
 179  
 180  If you pass the name of a script file as the first parameter, then it will output a link to only that file. Note that if it cannot find the script in any of the standard locations, it will not create a link for it.
 181  
 182  
 183      echo Assets::js('some_script.js');
 184  
 185  
 186  **Adding Scripts while Rendering**
 187  
 188  You can pass an array of script files as the first parameter of the script and they will be added to the scripts to be linked to. The function will then proceedd as if you had not passed any parameters in, creating links for all external, inline, and module scripts, including the ones that you did actually pass in. An optional second parameter allows you to specify whether the scripts are 'external' or 'inline';
 189  
 190  
 191      echo Assets::js( array('script1.js', 'script2.js'), 'inline' );
 192  
 193  
 194  
 195  <a name="external-js"></a>
 196  ### <tt>external_js()</tt>
 197  
 198  This method creates the links for all external scripts that it knows about. This is called automatically by the <tt>js()</tt> method and does not need to be called again if you are using that method.
 199  
 200  You can pass either an array of scripts or a single script as the first parameter and only those script files will have links created for them.
 201  
 202  If you are using a JS loader, like [head.js](http://headjs.com/), you might need a list of files instead of links created. You can get this by passing <tt>TRUE</tt> as the second parameter.
 203  
 204  
 205      echo Assets::external_js( array('script1.js', 'script2.js'), TRUE );
 206  
 207      // Produces
 208      "http://mysite.com/assets/js/script1.js", "http://mysite.com/assets/js/script2.js"
 209  
 210  
 211  By default, the Assets library will ensure that all files have a <tt>.js</tt> file extension. If you have some files have a different extension, you can pass FALSE as the third parameter.
 212  
 213  
 214  
 215  <a name="inline-js"></a>
 216  ### <tt>inline_js()</tt>
 217  
 218  Handles the inline script creation for the <tt>js()</tt> method. You do not need to call this script by itself when using that method.
 219  
 220  
 221  <a name="module-js"></a>
 222  ### <tt>module_js()</tt>
 223  
 224  Returns a link to a combined file that includes any scripts added with the <tt>add_module_js()</tt> method, below. When this method is called, all files are combined into a single file that is cached in the primary <tt>assets/cache</tt> folder.
 225  
 226  You can affect how the file is named as well as minification of the new file, via options that are discussed in Section 5, below.

 229  <a name="img-main"></a>
 230  ## Working With Images
 231  
 232  With mobile/desktop development becoming so prevalent, you often need a simple method to get an image that is only as big as you need it to be, smaller for mobile browsers and larger for desktops. With responsive design, you can fine-tune the bandwidth that your site's images are consuming even further based on the width of the browser window.
 233  
 234  Bonfire's image controller provides simple, on-the-fly creation of resized images. This is available at:
 235  
 236      http://mysite.com/images
 237  
 238  To return a square thumbnail of a large photo, you can choose to let the images library automatically crop the photo for you, while creating a thumbnail image. Pass along a <tt>size</tt> parameter that will define how wide and how tall the image should be.
 239  
 240  
 241      <a href="http://mysite.com/images?size=80">My Thumbnail</a>
 242  
 243  
 244  If you prefer to set a specific height and width, in pixels, for the image, you can use the <tt>height</tt> and <tt>width</tt> params to the url.
 245  
 246  
 247      <a href="http://mysite.com/images?height=60&width=100">My Thumbnail</a>
 248  
 249  You can also simply pass a <tt>width</tt> or <tt>height</tt> params to intelligently resize the image. Whichever param is passed will determine which is the "master" param. ex. If <tt>height</tt> is passed, the image will resize to a height of 60 and whatever width maintains aspect-ratio.
 250  
 251      <a href="http://mysite.com/images?height=60">
 252      
 253  
 254  By default, Bonfire will scale your image to fit within the new image size. To resize the image and disregard aspect-ratio, pass along a <tt>ratio=no</tt> parameter.
 255  
 256  
 257      <a href="http://mysite.com/images?size=80&ratio=no">My Thumbnail</a>
 258  
 259  
 260  Bonfire will look for the images within the <tt>assets/images</tt> folder in your web root. If you need to pull files from a different folder, you can use the <tt>assets</tt> parameter and pass in a folder relative to the webroot.
 261  
 262      <a href="http://mysite.com/images?size=80&assets=assets/images">My Thumbnail</a>

 265  <a name="performance"></a>
 266  ## Boosting Performance
 267  
 268  Large strides have been made over the last few years in recognizing techniques to help boost your client-side performance, or apparent performance to make it feel to the user like your site is loading faster. Bonfire provides a couple of different tools to help you with that.
 269  
 270  <a name="combining"></a>
 271  ### Combining Assets Into Fewer Files
 272  
 273  The first step to take usually involves combining your CSS and javascript files into as few files as possible to reduce the number of HTTP connections needed, since these are relatively slow.
 274  
 275  You can turn on combing files in your <tt>application.php</tt> configuration file. It is recommended to turn off combining in your development environment but turn it on in testing and production environments.
 276  
 277  
 278      $config['assets.combine'] = TRUE;
 279  
 280  
 281  This will create 2 files that are cached within the <tt>assets/cache</tt> folder. The first file combines all external files that are hosted within this site into a single file. The second file contains all of your module-specific files. This is the same file that is always generated for module js and css files.
 282  
 283  Combining works for both CSS and Javascript files.
 284  
 285  <a name="minimizing"></a>
 286  ### Minimizing Assets
 287  
 288  In addition to just combining the files, you can minimize your CSS and Javascript files, reducing the amount of bandwidth used for every page. You can turn on minizing files in your <tt>application.php</tt> configuration file. It is recommended to turn off minimizing in your development environment but turn it on in testing and production environments.
 289  
 290      $config['assets.js_minify'] = TRUE;
 291      $config['assets.css_minify'] = TRUE;
 292  
 293  
 294  
 295  
 296  <a name="encrypting"></a>
 297  ### Encrypting File Names
 298  
 299  While not a performance booster, those more security-conscious developers might dislike the typical naming convention of your cached asset files which display limited amount of information about the controller and module running. To obscure this information you can turn on file name encryption. This creates an md5 hash of the normal file name to cache the file as.
 300  
 301  
 302  
 303      $config['assets.encrypt_name'] = TRUE;
 304  
 305  
 306

 308  <a name="cache"></a>
 309  ### Clearing the Asset Cache
 310  
 311  You can delete all of your asset's cached files by calling the <tt>clear_cache()</tt> method. ALL cached assets will be deleted.
 312  
 313  
 314      Assets::clear_cache();

 317  <a name="helpers"></a>
 318  ## Helper Functions
 319  
 320  Three helper functions are provided to help create manual links to assets within your view files.
 321  
 322  <a name="assets_path"></a>
 323  ### <tt>assets_path()</tt>
 324  
 325  This returns a full URI to the main asset folder.
 326  
 327      echo assets_path();
 328  
 329      // Produces
 330      http://mysite.com/assets/
 331  
 332  
 333  <a name="css_path"></a>
 334  ### <tt>css_path()</tt>
 335  
 336  This returns a full URI to the CSS folder.
 337  
 338  
 339      echo css_path();
 340  
 341      // Produces
 342      http://mysite.com/assets/css/
 343  
 344  
 345  <a name="js_path"></a>
 346  ### <tt>js_path()</tt>
 347  
 348  This returns a full URI to the scripts folder.
 349  
 350  
 351      echo js_path();
 352  
 353      // Produces
 354      http://mysite.com/assets/js/
 355  
 356  
 357  <a name="img_path"></a>
 358  ### <tt>img_path()</tt>
 359  
 360  This returns a full URI to the images folder.
 361  
 362      echo img_path();
 363  
 364      // Produces
 365      http://mysite.com/assets/images/
