   1  ## What Are Contexts?
   2  
   3  A Context is an area of similar responsibility within the code and/or UI.  They appear as top-level navigation objects within the main menu, and should all cover a similar topic, or theme. The following are all good candidates for Contexts:
   4  
   5  Context         | Description
   6  ----------------|---------------
   7  Security        | Holds menu items from different modules that provide security tools and reports.
   8  Forums          | You can group all of your forums admin panels in a single context.
   9  Design          | You might provide methods for editing your templates or various error pages
  10  Tools           | Might contain collections of utilities that don't belong elsewhere.
  11  Role-Specific   | A single role might have access to a set of tools that no one else does. These could be grouped so they are easier to provide access to. |
  12  
  13  
  14  <a name="controllers"></a>
  15  ## Context Controllers
  16  
  17  Contexts map directly to controllers of the same name in your module. If you are creating a controller to handle your Settings Context for your module, the controller should be named <tt>content.php</tt> and the class should be named <tt>Content</tt>.
  18  
  19  
  20      class Content extends Admin_Controller {
  21          ...
  22      }
  23  
  24  <a name="views"></a>
  25  ## Context Views
  26  
  27  All views for your contexts should be stored within the views folder of your module, under a sub-folder named after the context. Continuing our example, the view for the index method would exist at <tt>my_module/views/content/index.php</tt>.
  28  
  29  
  30  <a name="public"></a>
  31  ## Public Contexts
  32  
  33  A Public Context is just a fancy name for what you would typically do in a CodeIgniter application. This controller is named the same as your module and can have as many methods that map directly to the URI as you need.
  34  
  35  Views are stored directly in your views folder. For front-facing page, or Public Contexts, there's little-to-no magic going on behind the scenes other than what you're already used to working with. The file would then be available at:
  36  
  37      http://yoursite.com/module_name
  38  
  39  
  40  <a name="required"></a>
  41  ## Required Contexts
  42  
  43  Bonfire requires only two contexts to exist, and will create them if they don’t. They are the **Developer** and **Settings** Contexts.
  44  
  45  Context     | Description
  46  ------------|------------------
  47  Developer   | This context is meant for tools designed to make the application developer’s life easier and may include database tools, code generation, deploy helpers, etc. |
  48  Settings    | Any settings related to your module.  These are typically items that are infrequently changed, like the number of events to show on each page, etc. |
  49  
  50  
  51  <a name="custom"></a>
  52  ## Creating New Contexts
  53  
  54  You can create any number of contexts to fit your application’s specific needs.  Doing so is a very simple task that only requires a single step.
  55  
  56  <a name="array"></a>
  57  ###  Update the Application Config Array
  58  
  59  To create a new context that will be displayed in the admin area, you only need to add the name of the context to the <td>contexts</td> array in the <td>config/application.php</td> file. The context name must not contain any spaces. If more than one word is needed, separate the words with an underscore.
  60  
  61  
  62  <a name="name"></a>
  63  ### Localizing the Context Name
  64  
  65  To make your context's name localized, you simply need to add a new entry in the <td>language/english/application_lang.php</td> file. The name should be preceded by <td>bf_context_</td>.
  66  
  67  
  68      $lang['bf_context_new'] = 'New Context';
  69  
  70  
  71  The localized version will automatically be used if present, otherwise the name displayed will be the name of the context.
  72  
  73  <a name="access"></a>
  74  ### Restricting Access to Contexts
  75  
  76  You can restrict access to your contexts by creating a new permission. This permission should be named 'Site.{context}.View', where <td>{context}</td> is the name of your context. Once this permission is created in the system, no one will be able to view the context if they do not have permission.
  77  
  78  In order to manage the permission, you will need to create another permission called 'Permissions.{context}.Manage'.
